# 模型选择策略

VCDDD 的子 Agent 派遣使用分层模型策略。

---

## 模型角色定义

VCDDD 使用两个抽象模型角色，不硬编码具体模型名称：

| 角色 | 定位 | 用途 |
|------|------|------|
| **效率模型** | 速度快、成本低、适合常规任务 | 代码实现、测试编写、审查、TDD Bridge、集成验证 |
| **深度思考模型** | 推理深、能力强、成本高 | 升级排查（多次失败后的根因分析 + 修复方案） |

---

## 第零步：环境识别

**在进入 D2-auto 之前，主控必须先将抽象角色映射到具体模型名称。**

### Claude 环境

Agent 工具支持以下预定义别名：

| 角色 | 别名 | 对应模型 |
|------|------|---------|
| 效率模型 | `sonnet` | Claude Sonnet |
| 深度思考模型 | `opus` | Claude Opus |
| 轻量任务 | `haiku` | Claude Haiku（格式检查、简单文档生成） |

**识别方法**：检查当前 session 是否在 Claude Code 中运行。如果是，直接使用上述别名。

### 其他环境（GPT、本地模型、第三方 API 等）

**必须询问用户：**

```
当前环境：{检测到的环境}
VCDDD 需要为子 Agent 指定模型。VCDDD 使用两个模型角色：
  - 效率模型：用于代码实现、测试编写、审查等常规任务
  - 深度思考模型：用于多次失败后的根因排查

请告知：
1. 你希望用作「效率模型」的具体模型名称（如 gpt-4o-mini）
2. 你希望用作「深度思考模型」的具体模型名称（如 gpt-4o）
3. 推理长度/思考预算（如适用）
```

### 无法识别时

如果环境检测失败或不确定，**直接询问用户**，不猜测。

---

## 第一步：变量校验

获取到模型名称后，必须校验：

```
获取模型名称（来自环境检测或用户输入）
    │
    ├── Claude 环境
    │     → 检查别名是否为：sonnet / opus / haiku
    │     → 无效别名 → 询问用户
    │
    └── 其他环境
          → 使用用户提供的名称
          → 格式检查：名称不能为空、不含特殊字符
```

**禁止猜测模型名称。** 用户说叫什么就用什么。

---

## 第二步：分层模型分配

| 任务类型 | 模型角色 | 说明 |
|---------|---------|------|
| TDD Bridge Generator | 效率模型 | 测试规格推导，规则驱动 |
| TDD Bridge Reviewer | 效率模型 | 测试规格审查 |
| Implementer | 效率模型 | 代码实现 |
| Spec Reviewer | 效率模型 | 规格合规审查 |
| Quality Reviewer | 效率模型 | 代码质量审查 |
| VCDDD Reviewer | 效率模型 | VCDDD 合规审查 |
| 集成验证 Agent | 效率模型 | 跨域集成验证 |
| 设计文档编写 | 主控 Session | business.md、boundary.md、tech-stack.md |
| **升级排查** | **深度思考模型** | **多次失败后的根因分析** |

---

## 第三步：升级机制

### 触发条件

**同一类型的问题**在多轮修复中反复出现，效率模型无法解决时触发升级。

**区分两种情况**：

| 情况 | 说明 | 处理 |
|------|------|------|
| 每轮都是**新的问题** | 第 1 轮修了 A，第 2 轮引入了 B，第 3 轮引入了 C | 正常迭代，继续用效率模型 |
| **同一问题**反复修不好 | 第 1 轮修 A 没修好，第 2 轮修 A 还是没修好，第 3 轮 A 依然存在 | 升级到深度思考模型 |

**判断方法**：对比 Reviewer 每轮返回的 ISSUES 列表。如果连续 **2 轮**存在相同的未解决问题（相同的错误描述、相同的测试失败、相同的代码位置），则触发升级。

### 升级流程

```
Reviewer 返回 ISSUES，控制器发现：
  → 本轮问题 X 与上一轮的问题 X 相同（未被修复）
  → 连续 2 轮同一问题 X 反复出现
    │
    ▼
  停止当前效率模型循环
    │
    ▼
  用深度思考模型派遣升级排查 Agent：
    传入：
      - 该域全部文档（business.md + boundary.md + tech-stack.md）
      - Implementer 的全部代码
      - Reviewer 的全部审查记录（多轮问题列表，标注哪些问题重复出现）
      - Implementer 的多轮修复记录
    任务：
      - 分析问题 X 为什么反复出现（是理解偏差？是实现方向错误？是文档矛盾？）
      - 给出具体的修复方案
    │
    ▼
  排查 Agent 返回诊断报告 + 修复方案
    │
    ▼
  回到效率模型，派遣 Implementer 按方案修复
    │
    ▼
  重新派遣 Reviewer 审查
```

### 升级排查 Agent 的约束

- **只诊断，不修改代码** — 分析根因并给出方案，不自己动手改
- **方案必须具体** — 给出"改哪几个文件的哪几行"，不是"建议检查一下"
- **诊断完成后回到效率模型** — 修复由效率模型的 Implementer 执行

---

## 控制器的模型使用指令

在每次派遣子 Agent 时，控制器必须：

1. 从模型配置中读取该任务类型的角色（效率模型 / 深度思考模型）
2. 将角色映射为具体模型名称
3. 在 Agent 工具调用时传入 `model` 参数

示例（Claude 环境）：
```
# 效率模型派遣
Agent(task="...", model="sonnet")

# 深度思考模型派遣（升级）
Agent(task="...", model="opus")
```
