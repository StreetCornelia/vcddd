# 跨域集成验证

跨域集成验证在 D3-auto 流程中扮演**系统级对抗方**的角色。

**定位**：它不是被审核的对象，而是审核各域实现的人在跨域协作中是否一致的对抗方。它的视角不是"这段代码对不对"，而是"这些域拼在一起，业务上是否说得通"。

---

## 角色设定

集成验证的职责是：

1. 基于 `facts.md` 中的业务全景和各域的 `business.md`，检查跨域的协作是否正确
2. 发现问题后，区分两类情况并分别处理：
   - **技术层面的问题**：直接交给编排者协调修复，不涉及用户
   - **业务层面的问题**：基于业务理解给出建议方案，与用户交互并细化业务文档
3. 最终确保各域的实现在业务语义上能够正确配合

---

## 工作流程

```
各域实现完成 + 通过各自的三层审查
        │
        ▼
  [识别问题]
  运行全部验证检查
  对每条检查结果：
    ✅ 符合标准 → 通过
    ❌ 不符合标准 → 判断问题类型
        │
    ┌───┴───────────────┐
    │                   │
  [技术问题]           [业务细化问题]
  字段不匹配           事件顺序未定义
  幂等缺失             并发冲突规则
  依赖方向错误         错误恢复策略
  类型不一致           超时处理未明确
    │                   │
    ▼                   ▼
  提交给编排者          向用户呈现：
  → 编排者派遣          · 当前场景描述
    对应域               · 基于业务理解给出的
    Implementer            建议方案
    直接修复            · 影响范围
                        · 用户决策后更新
                          business.md
    │                   │
    └───────────┬───────┘
                ▼
        重新执行集成验证
        （最多 10 轮对抗）
```

---

## 两类问题的处理

### 技术问题：不需要用户参与

**特征**：可以从现有文档（boundary.md / tech-stack.md）直接推导出正确的修复方案，不改变业务含义。

**处理方法**：编排者收到问题列表后，直接派遣对应域的 Implementer 修复，修复后重新验证。

```
集成验证：发现 payment/OrderPlaced 的 orderId 字段名不一致
  → 提交编排者
编排者：派遣 payment 域的 Implementer 修复字段命名
Implementer：修复 → 提交
编排者：重新运行集成验证
```

这类问题不打断用户。

### 业务细化问题：需要用户确认 + 触发域级重实现

**特征**：现有 `business.md` 未覆盖的业务场景，修复需要新增或修改业务规则。

**完整对抗循环**：

```
集成验证发现业务未定义场景
        │
        ▼
  基于业务理解给出建议方案
  暂停执行，向用户呈现：
    1. 场景描述
    2. 建议方案 + 理由
    3. 影响范围（涉及哪些域）
        │
        ▼
  用户回复：
    a. 接受建议
    b. 自己给方案
    c. 暂时跳过 → DONE_WITH_CONCERNS
        │
   (a/b) ▼
  更新涉及域的 business.md
        │
        ▼
  受影响域重新进入域级对抗循环：
    ① Implementer 按新 business.md 重新实现
    ② Spec Reviewer → Quality Reviewer → VCDDD Reviewer 三层审查
    ③ 通过后回到集成验证
        │
        ▼
  重新运行集成验证（检验修改是否解决了问题）
  如果仍有问题 → 重复上述循环（最多 10 轮）
```

**示例**：

```
集成验证发现：
  场景：CancelOrder 与 PaymentCompleted 同时到达同一笔订单
  business.md 现状：未定义优先级

建议方案：
  CancelOrder 优先于 PaymentCompleted。
  理由：先处理 PaymentCompleted 后再 CancelOrder 会产生"已支付但取消"
  的账单状态，需要额外的退款流程。CancelOrder 优先可避免不必要的支付处理。

影响域：order（修改状态机冲突处理规则）

用户：接受建议

  → 更新 order/business.md：在状态机中增加冲突优先级规则
  → 派遣 order Implementer：按新 business.md 重新实现状态机
  → order 三层审查：Spec → Quality → VCDDD
  → 集成验证：重新运行，确认问题解决
```

**关键**：
- 集成验证给出的是**建议**，不是决定。在用户确认之前，不写入 business.md
- `business.md` 的变更会触发该域的完整重实现 + 三层审查，因为业务规则的改变可能影响代码逻辑
- 整个集成验证对抗循环最多 10 轮（含重新实现的时间）

## 验证检查

### 1. 契约结构验证

跨域事件/命令的字段、类型是否一致。

```
OrderPlaced 事件（order → payment, inventory）
  boundary.md 定义：{ orderId, userId, items, totalAmount, timestamp }
  
  检查 order 域的发布端：
    [ ] 字段齐全，类型正确
  检查 payment 域的消费端：
    [ ] 字段名一致
    [ ] 类型一致
    [ ] 必填项未忽略
  检查 inventory 域的消费端：
    [ ] 字段名一致
    [ ] 类型一致
```

### 2. 端到端工作流验证

从 facts.md 的核心业务流程出发，检查整条链路是否通畅。

```
流程：用户下单 → 支付 → 发货

[order]  PlaceOrder 调用成功
  → [order] 发布 OrderPlaced
    → [payment] 消费 OrderPlaced → 发起支付 → 成功
      → [payment] 发布 PaymentCompleted
        → [order] 消费 PaymentCompleted → 状态变 CONFIRMED
        → [inventory] 消费 OrderPlaced → 库存预占

每个节点的产出与下一节点的输入是否匹配：
  OrderPlaced 的字段能否满足 payment 发起支付所需的信息？
  PaymentCompleted 的字段能否让 order 确认是哪笔订单？
```

### 3. 幂等验证

```
PaymentCompleted 重复发送：
  order 域消费两次 → 结果与消费一次是否相同？
  → 状态不重复变更
  → 库存不重复扣减
  → 日志区分"首次处理"和"幂等命中"
```

### 4. 事件顺序与并发

```
场景：PaymentCompleted 先于 OrderPlaced 到达 payment 域
  · business.md 是否定义了这种行为？
  · 如果未定义 → 业务细化问题，需用户确认

场景：CancelOrder 与 PaymentCompleted 同时到达
  · 两笔操作涉及同一笔订单的不同方向
  · business.md 是否定义了优先级？
  · 如果未定义 → 业务细化问题，需用户确认
```

### 5. 依赖方向验证

```
检查跨域引用关系与 boundary.md 的上下文映射一致：
  [ ] order 域没有直接导入 payment 域的内部模块
  [ ] payment 域没有直接访问 order 域的数据库
  [ ] 所有跨域通信都通过事件机制
  [ ] 没有循环依赖
```

## 编排者（总控）的协调逻辑

集成验证作为对抗方只负责"发现问题 + 给出建议"。实际的问题处理和流程推进由编排者（总控节点）执行。

### 技术问题的协调

```
编排者收到集成验证的技术问题列表
  │
  ├── 问题 A（字段名不一致，域 X）
  │     → 派遣域 X 的 Implementer 修复
  │     → Implementer 修复后提交
  │
  ├── 问题 B（幂等缺失，域 Y）
  │     → 派遣域 Y 的 Implementer 修复
  │     → Implementer 修复后提交
  │
  └── 全部修复完成后
        → 重新运行集成验证
```

技术问题的修正范围小（改字段名、加幂等检查），不需要重新过三层审查。编排者可以直接确认修复有效。

### 业务细化问题的协调

```
编排者收到集成验证的业务细化问题列表
  │
  ├── 向用户呈现（暂停自动化）
  │
  ├── 用户接受建议
  │     → 更新涉及域的 business.md
  │     → 派遣各域的 Implementer 重新实现
  │     → 各域重新过三层审查（Spec → Quality → VCDDD）
  │     → 重新运行集成验证
  │
  ├── 用户自己给方案
  │     → 按用户方案更新 business.md
  │     → 派遣各域的 Implementer 重新实现
  │     → 各域重新过三层审查
  │     → 重新运行集成验证
  │
  └── 用户选择跳过
        → 标记为 DONE_WITH_CONCERNS
        → 在最终报告中列出未解决问题
        → 不阻塞后续流程
```

**关键**：业务细化问题更新了 `business.md`，这意味着域的规格发生了变化。因此涉及域的代码必须**重新走过完整的域级对抗循环**（Implementer + 三层审查），不能只做局部修补。这与技术问题的处理有本质区别。
