# TDD 桥梁：从 business.md 推导黑盒测试规格

从已确认的 `business.md` 和 `boundary.md` 中，推导域的**黑盒功能测试规格**。

本文档的定位是**参考文件**，与 `reference/engine/` 下其他文档并列。实际使用时由编排者按规则机械推导。

## 核心原则

### 原则零：使用自然语言或伪代码，不写具体代码

测试规格的目的是**描述业务的期望行为**，不是约束实现方式。

- 使用自然语言或伪代码描述测试场景和预期结果
- **不写具体编程语言的代码**（不写 assert、expect、jest 等语法）
- 不假设实现的语言、框架、测试工具

```
✅ 正确（伪代码/自然语言）：
  测试：已发货订单执行撤销
    → 前置条件：订单状态 = SHIPPED
    → 操作：执行撤销命令
    → 预期：返回业务拒绝，原因 "订单已发货，不可撤销"
    → 错误来源：Order 聚合的不变式校验

❌ 错误（具体代码，限制实现）：
  test('shipped order cannot be cancelled', () => {
    expect(() => order.cancel()).toThrow('订单已发货')
  })
```

**原因**：用具体代码编写测试会限制后续实现 Agent 的发挥空间。伪代码/自然语言描述的是"业务上期望什么"，具体代码描述的是"代码应该怎么写"。前者给实现留出了自由，后者把实现锁死在了测试作者的写法上。

### 原则一：仅黑盒测试

TDD 桥梁的产出**只测试域对外提供的能力**，不定义内部实现方式。

- 测试针对的是 `boundary.md` 中定义的命令入口、事件消费接口、读模型查询
- 不测试内部聚合、仓储实现、私有方法
- 不假设内部代码结构（不假设用了什么类、什么设计模式）

内部实现的测试（白盒、单元测试等）由后续 Code Implementation 阶段的 Subagent 自行完成，并通过 Quality Reviewer 把关。

### 原则二：全覆盖，不抽样

对域对外能力的每条分支、每个边界、每种错误情况，必须有对应的测试用例。

三个覆盖维度：

| 维度 | 含义 | 验证目标 |
|------|------|---------|
| **功能完整性** | 域对外承诺的每条能力是否有对应测试 | 每条命令、事件、读模型都有覆盖 |
| **分支完整性** | 每条命令的所有正常路径 + 失败路径是否有对应测试 | business.md 列出的每条路径都有覆盖 |
| **数据边界性** | 可能导致业务逻辑错误的数据输入是否有对应测试 | 非法输入进入错误分支，且错误处理语义一致 |

### 原则三：目标必须详细，其余可推演

每条测试用例的核心是**目标**——它描述这个测试要验证的业务场景、业务原因和预期业务结果。

目标是测试用例中**最关键的字段**，必须写得充分具体。其余字段（前置条件、操作、期望、错误来源等）是目标的环境支撑，如有遗漏，实现 Agent 可以基于目标的业务描述自行推演补充。

```
✅ 目标充分：
  目标：验证已发货的订单不能被撤销。业务上已发货代表履约流程已不可逆，
       撤销操作应被拒绝，防止产生"已发货但取消"的矛盾状态。
  前置：订单状态 = SHIPPED
  操作：执行撤销命令
  期望：拒绝，原因 "订单已发货，不可撤销"
  错误来源：Order 聚合的不变式校验

✅ 目标充分：
  目标：验证 orderId 为空字符串时 PlaceOrder 被拒绝。订单标识符是唯一标识
       一笔交易的业务ID，空字符串是无意义的输入，必须被拦截在聚合外。
  期望：拒绝，原因 "订单ID不能为空"
```

**原因**：充分描述的目标使得实现 Agent 即使在其他字段不完整的情况下，也能基于业务理解自行推导出正确的测试实现。这给实现留出了推演空间，不把实现锁死在固定格式里。

### 原则四：一致性检查

如果 business.md 中多条失败路径的错误处理方式（如错误码结构、错误分类、返回值格式）不一致，标记为 `[一致性检查]` 并在 test-spec.md 中标注。

## 前置条件

- `docs/vcddd/design/{domain}/business.md` 已存在且经用户确认
- `docs/vcddd/design/{domain}/boundary.md` 已存在

## 推导流程

```
business.md + boundary.md
        │
        ▼
  1. 功能完整性扫描 → 域对外有哪些能力？
  2. 分支完整性扫描 → 每条能力有哪些路径？
  3. 数据边界扫描   → 每个输入参数有哪些边界情况？
        │
        ▼
  按规则逐条推导测试用例（全覆盖，不抽样）
        │
        ▼
  test-spec.md（可追溯至 business.md 每行）
```

## 规则说明

### 功能完整性扫描

扫描 `boundary.md`，识别域对外暴露的命令、事件、读模型：

```
boundary.md 中定义的：
  命令（Commands）：       域可以接收哪些业务意图
  事件（Events）：         域发布哪些业务事实、消费哪些事件
  读模型（Read Models）：  域提供哪些查询能力

每个命令/事件消费入口/读模型 → 至少一个正常路径测试
```

这条规则确保**域对外承诺的每条能力都被测试覆盖**。

### 分支完整性扫描

对每条命令/事件消费入口，扫描 business.md 中定义的路径：

```
正常路径：1 条 → 1 个正常结果测试
失败路径：N 条 → N 个对应错误测试（逐条覆盖，不遗漏）
幂等路径：1 条 → 1 个重复执行测试

blanket rule: business.md 列出的每条路径必须有对应测试用例
```

分支完整性只关注 terminal 判定点（即直接产生成功或失败结果的分支逻辑）。数据校验等由数据边界性覆盖。

### 数据边界性扫描

对命令/事件消费入口的每个参数，扫描边界情况：

```
参数类型          边界情况
─────────────────────────────────────────────
字符串            null / 空字符串 / 超长 / 非法格式（如 email 格式错误）
数值              0 / 负数 / 超过业务上限 / 浮点精度
枚举              不在枚举范围内的值
列表              空列表 / 包含非法元素的列表 / 超长列表
可选字段          null 是否可接受
关联 ID           不存在的 ID / 已删除的 ID / 格式错误的 ID
```

每个参数边界产生至少 1 个错误分支测试 + 对应的错误原因。

## 推导示例

### business.md 输入

假设 `docs/vcddd/design/order/business.md` 定义了：

```
命令：
  PlaceOrder(orderId, userId, items, totalAmount)

正常路径：
  创建订单 → 扣减库存预占 → 发布 OrderPlaced 事件

失败路径：
  F1: userId 不存在        → "用户 {userId} 不存在"
  F2: items 为空           → "订单必须包含至少一件商品"
  F3: items 中包含下架商品  → "商品 {productId} 已下架"
  F4: 库存不足             → "商品 {productId} 库存不足 (可用: {n})"
  F5: 重复 orderId         → 幂等返回首次结果

幂等策略：
  幂等键：orderId，重复时返回相同结果，不重复执行
```

### 推导结果

```
CMD-001: [正常] 有效参数 → 订单创建成功，库存预占扣减，OrderPlaced 发布
CMD-E01: [失败] userId 不存在 → 返回 "用户 xxx 不存在"，无副作用
CMD-E02: [失败] items 为空列表 → 返回 "订单必须包含至少一件商品"
CMD-E03: [失败] items 包含已下架商品 → 返回 "商品 yyy 已下架"
CMD-E04: [失败] items 中某个商品库存不足 → 返回 "商品 yyy 库存不足 (可用: 0)"
CMD-IDM-01: [幂等] 相同 orderId 重复请求 → 返回与第一次相同的结果，不重复扣减库存

边界测试：
CMD-B01: [边界] totalAmount 为 0 → 返回错误（依赖业务规则，如 "订单金额必须大于 0"）
CMD-B02: [边界] totalAmount 为负数 → 返回错误（无意义的业务输入）
CMD-B03: [边界] orderId 为空字符串 → 返回错误（标识符不能为空）
CMD-B04: [边界] items 包含 quantity=0 → 返回错误（数量必须为正数）
```

注意：边界测试中 `totalAmount=0` 是否合法取决于 business.md。如果 business.md 没有约束 totalAmount，则在 test-spec 中标注 `[待确认]`。

### 一致性检查

如果两条失败路径的错误返回格式不同，如：

```
F1: 返回 { errorCode: "USER_NOT_FOUND", message: "用户 xxx 不存在" }
F3: 返回 "商品已下架"  // 纯字符串，没有 errorCode
```

→ 标注：`[一致性检查] CMD-E01 和 CMD-E03 的错误返回格式不一致，需确认`

## 完整性检查清单

完成推导后，逐项核对：

### 功能完整性
- [ ] 每条命令都有覆盖（正常路径 + 全部失败路径 + 幂等）
- [ ] 每个事件消费入口都有覆盖
- [ ] 每个读模型查询都有覆盖

### 分支完整性
- [ ] business.md 每条命令的正常路径 → 1 个测试
- [ ] business.md 每条命令的每条失败路径 → 1 个测试（逐条，不合并）
- [ ] 幂等策略 → 1 个幂等测试

### 数据边界性
- [ ] 每个字符串参数 → 空字符串/超长/格式非法
- [ ] 每个数值参数 → 0/负数/超上限
- [ ] 每个枚举参数 → 非法值
- [ ] 每个列表参数 → 空列表/非法元素
- [ ] 每个关联 ID → 不存在的 ID/格式错误

### 一致性
- [ ] 所有错误路径的返回格式一致（都有 errorCode + message 等）
- [ ] 相同语义的错误码跨路径一致

## business.md 遗漏处理

推导过程中发现 business.md 有遗漏（如某条命令的失败路径没列全、状态机非法迁移没定义、边界条件未覆盖），**不自行补充**，标记为 `[待补充]` 并列出缺失项，在进入实现前交用户确认。

## 对抗生成流程

test-spec.md 的生成和验证通过两个 subagent 的对抗循环完成：

```
编排者准备上下文
（该域 business.md + boundary.md + tdd-bridge.md 规则）
        │
        ▼
[Subagent 1] Test Spec Generator
  根据规则机械推导 test-spec.md（全覆盖，不抽样）
        │
        ▼
[Subagent 2] Test Spec Reviewer
  独立审查生成的 test-spec.md
        │
    ┌───┴───┐
    │ 审查通过？ │
    └───┬───┘
   否   │  是
    ▼   │   ▼
  Generator    输出最终的
  根据反馈       test-spec.md
  修正         写入文档
    │
    ▼
  重新 Review
  (最多 10 轮)
```

### Generator 的输入上下文

编排者传递给 Generator subagent 的内容：

```
## 域文档
{domain}/business.md（全文）
{domain}/boundary.md（全文）

## 推导规则
{tdd-bridge.md 的规则说明}

## 要求
1. 黑盒测试：只测域对外能力（命令入口、事件消费、读模型），不测内部实现
2. 全覆盖：功能完整性 + 分支完整性 + 数据边界性，不抽样
3. 每条失败测试必须有具体错误原因
4. 执行一致性检查，不一致的标记出来
5. business.md 未定义的内容标记 [待补充]，不自行假设
6. 按模板写入 docs/vcddd/design/{domain}/test-spec.md
```

### Reviewer 的检查清单

编排者传递给 Reviewer subagent 的内容：

```
## 审查目标
核对生成的 docs/vcddd/design/{domain}/test-spec.md 是否满足以下要求：

### 1. 是否符合整体要求和业务要求

- [ ] 所有测试都是黑盒测试（没有引用内部类、内部方法、私有状态）
- [ ] 测试只针对域对外能力（boundary.md 定义的命令/事件/读模型）
- [ ] 每条测试用例的业务语义与 business.md 一致

### 2. 内容是否完整覆盖了业务可能的分支

- [ ] 功能完整性：每条命令/事件消费/读模型至少有一个正常路径测试
- [ ] 分支完整性：business.md 列出的每条失败路径都有对应测试（逐条核对，不遗漏）
- [ ] 分支完整性：幂等策略有对应测试
- [ ] 业务中的潜在缺失已标注 [待补充]

### 3. 错误或正确的情况是否满足业务规定

- [ ] 每条正常路径的预期结果与 business.md 的描述一致
- [ ] 每条失败路径的错误原因与 business.md 的描述一致（逐字逐句核对）
- [ ] 每条数据边界测试的错误原因无语义冲突

### 4. 处理其他潜在的可能问题

- [ ] 错误返回格式在跨路径间一致（都有 errorCode + message + 分类）
- [ ] 数据边界测试覆盖了所有参数的边界情况（空/负/超长/非法枚举/不存在 ID）
- [ ] business.md 未定义的边界没有自行补充（标记为 [待补充]）
- [ ] 一致性检查已执行，不一致处已标注

### 判定

✅ 全部通过 → 写入 test-spec.md
❌ 存在问题 → Generator 修复后重新审查

### 待修正项

列出具体不符合项（引用 test-spec.md 中的具体行号和测试 ID）：
1. ________
2. ________
```

### 轮数限制

Generator ↔ Reviewer 循环遵循全局对抗生成规则（详见 `reference/engine/review-loop.md` 的"通用对抗生成框架"）：
- 最多 10 轮
- Reviewer 必须一次性给出全部修改意见
- 超过 10 轮后按升级流程处理（拆分或标记 DONE_WITH_CONCERNS）
