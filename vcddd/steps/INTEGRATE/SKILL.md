> 子 Agent 指令：本文件为子 Agent 的完整执行指令，由总控 Agent 传入并派遣执行。
> 本文件同时是控制器的调度参考——控制器必须按以下调度循环执行，不需要用户提醒。

# Step: INTEGRATE — 跨域集成验证

验证各域实现之间的协作是否一致。这是系统级对抗方，检查多域的业务语义拼合。

## 前置条件

- 所有域的代码已实现并通过各自的三层审查

## 输入

- `facts.md`（业务全景）
- 各域 `business.md` + `boundary.md`
- 各域实现代码

## 输出

跨域集成验证报告

## 控制器传给集成验证 Agent 的上下文

```
## 任务
你是跨域集成验证 Agent。验证各域实现之间的协作一致性。

## 业务全景
{facts.md 全文}

## 各域业务设计
{逐域列出 business.md 全文}

## 各域边界设计
{逐域列出 boundary.md 全文}

## 各域实现代码
{逐域列出全部代码文件}

## 验证检查项
1. 契约结构：跨域事件/命令的字段、类型是否一致
2. 端到端工作流：从事实中的核心业务流程出发，检查链路是否通畅
3. 幂等：跨域命令重复执行结果是否一致
4. 事件顺序与并发：未定义的事件到达顺序是否标注
5. 依赖方向：无反向依赖，无循环依赖

## 输出要求
返回以下格式的验证报告：
- PASS（全部检查通过）
- TECH_ISSUES（技术问题列表：字段不匹配/幂等缺失/依赖方向错误/类型不一致）
- BIZ_ISSUES（业务细化问题列表：事件顺序未定义/并发冲突规则/错误恢复策略未明确）
- MIXED（两者都有）

每个问题必须包含：问题描述、涉及域、建议修复方式。
```

## 调度循环（控制器强制执行）

```
┌─ 集成验证循环开始 ─────────────────────────────────────┐
│                                                         │
│  1. 按上方模板派遣集成验证 Agent                          │
│     → 返回：PASS / TECH_ISSUES / BIZ_ISSUES / MIXED    │
│                                                         │
│  2. 返回 PASS 后：                                       │
│     → 集成验证完成，控制器进入 REPORT 步骤                │
│                                                         │
│  3. 返回 TECH_ISSUES 后【立即】：                         │
│     → 不打断用户                                         │
│     → 控制器对每个技术问题：                              │
│       · 判断属于哪个域                                    │
│       · 派遣该域的 Implementer，传入：                    │
│         - 足够多的背景信息（涉及的事件/命令定义）          │
│         - 问题列表 + 建议修复方式                         │
│       · Implementer 修复后返回                            │
│     → 全部修复后【立即】重新派遣集成验证 Agent（回到 Step 1）│
│     → 循环直到 PASS 或达到 10 轮上限                     │
│                                                         │
│  4. 返回 BIZ_ISSUES 后【立即】：                          │
│     → 暂停自动化                                         │
│     → 向用户呈现场景：                                    │
│       · 当前问题描述                                      │
│       · 基于业务理解给出的建议方案                         │
│       · 影响范围                                          │
│     → 等待用户决策                                        │
│     → 用户确认后更新 business.md                          │
│     → 受影响域重新过 REVIEW-DOMAIN                        │
│     → 重新派遣集成验证 Agent（回到 Step 1）               │
│                                                         │
│  5. 返回 MIXED 后【立即】：                               │
│     → 先处理 TECH_ISSUES（Step 3，不打断用户）            │
│     → 再处理 BIZ_ISSUES（Step 4，打断用户）              │
│     → 全部处理后重新派遣集成验证 Agent（回到 Step 1）      │
│                                                         │
└─ 集成验证循环结束 ─────────────────────────────────────┘
```

**关键约束**：
- **技术问题不打断用户**：控制器直接协调 Implementer 修复
- **业务问题必须打断用户**：涉及业务决策，不能 AI 自行决定
- **修复后必须重验**：任何修复后都必须重新运行集成验证
- **同一个验证 Agent 检查修复结果**：技术问题修复后，让同一个验证 Agent 重新检查
- **不需要用户提醒**：技术问题的修复循环是自动的

## 如果卡住

读 `../../reference/engine/integration-verification.md` 获取完整验证清单。

## 日志

在 `docs/vcddd/progress.log` 追加 Task 节点。记录每项检查结果和协调修复操作。

- [ ] 业务细化问题已向用户呈现并得到回复
- [ ] 修改 business.md 后对应域已重新通过审查
- [ ] 对抗循环不超过 10 轮
