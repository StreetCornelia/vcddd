# Step: D¹ — Domain Design（域设计）

仅以确认的业务事实为输入，推导域边界、决策归属、不变式、事件与协作契约。

## 前置条件

- `docs/vcddd/requirements/facts.md` 已存在且经用户确认

## 输入

- `facts.md`（唯一的输入来源）

## 输出

每个域：`docs/vcddd/design/{domain}/boundary.md` + `business.md`

## 执行流程

**每个域设计完立即向用户确认，确认后立即写入。不等全部域设计完再一起确认。**

### 第一步：建立全局视图

读 facts.md，建立业务事实全景。

### 第二步：逐域推导（每域独立确认）

对每个域，按以下顺序推导：

1. **识别域边界**：对每组事实追问——独立语言？独立不变式？独立演进？
2. **设计决策主权**：每个域记录哪些判断只能由它做出
3. **设计不变式**：每条必须有违反后果
4. **设计状态机**：合法迁移 + 非法迁移
5. **设计协作契约**：命令、事件、失败语义

**该域推导完成后，立即向用户展示该域的 boundary.md + business.md，等待用户确认。**

### 第三步：发现 facts.md 歧义时

设计过程中发现 facts.md 有歧义（两种理解都说得通）时：

1. **不自行假设**
2. 给出几个可能的理解方向，每个附带置信度（如"方向 A（置信度高）：...，方向 B（置信度中）：..."）
3. 让用户选择或描述自己的理解
4. 用户确认后更新 facts.md，再继续设计

### 第四步：写入文档

用户确认后，立即写入 `docs/vcddd/design/{domain}/boundary.md` 和 `business.md`。

## 禁止

- 讨论数据库表结构
- 从技术模块反推域边界
- 把"调用关系"当作"协作契约"

## 如果卡住

读 `../../reference/guides/design.md` 获取完整流程。

## 日志

创建 `docs/vcddd/design/{domain}/progress.log` Task 1。每个域边界识别、每写入一个文档 → 追加记录。

## 验证

- [ ] 每个域能力边界明确
- [ ] 每条不变式有违反后果
- [ ] 状态机含合法和非法迁移
- [ ] 每个命令全部失败路径已列出
- [ ] 每个域经用户确认后写入文档
