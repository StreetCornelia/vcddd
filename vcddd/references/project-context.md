## 工作目的

使没有历史对话的人或 AI 先恢复项目全局认识和进行中任务导航，在用户明确选择后再进入中文任务；同时支持只读了解、一个对话推进多个任务和多个对话并行工作，并避免把全部历史重新装入上下文。

将目标仓库中的 `docs/vcddd/` 作为项目专属共同认知空间。将 Skill 自身的 `references/` 视为跨项目行动规则；不要混写两者。

## 判断变量

先判断：

- `docs/vcddd/index.md` 是否存在，并能否指向业务入口、任务导航和必要材料；
- `docs/vcddd/work/index.md` 列出了哪些进行中、已完成或已取消任务；
- `docs/vcddd/systems/index.md` 是否存在，项目是否需要多系统、模块或跨系统关系认识；
- 本次请求属于哪个任务，还是形成了具有独立完成条件的新任务；
- 哪些内容是当前权威业务或系统认识、当前假设、潜在线索、历史或已替代内容；
- 当前问题涉及哪些业务线、系统或模块，是否需要理由、证据或历史任务才能判断；
- 文档、图、原型、代码和运行行为之间是否冲突；
- 多个进行中任务是否同时影响同一业务、模块领域或设计并产生冲突；
- 本次变化是否会成为跨任务继续成立的业务或系统认识；
- 当前请求是否允许写入项目文件：持续构建、修改或迭代通常包含认知维护，单纯解释、审查或报告则保持只读。

## 选择行动

### 首次建立项目档案

适用于用户要求从当前材料梳理整个项目，并建立后续可以逐步完善的 VCDDD 认知空间。

- 将“建立初步项目认知”作为一个中文工作任务，而不是把发现的每条业务目标线直接创建成任务。
- 先按 [requirement-discovery.md](requirement-discovery.md) 从现有材料形成现实背景、总体目标、主要大目标、收益、边界、不可接受结果和候选遗漏，再按 [business-understanding.md](business-understanding.md) 建立需求目标—业务目标线覆盖、业务目标线清单和前提依赖；不要从最先发现的功能直接下钻，也不要先建空目录等待填写。
- 新项目优先建立 `docs/vcddd/index.md` 作为唯一入口、`docs/vcddd/business/index.md` 作为需求全景与业务全貌入口，并在 `docs/vcddd/work/index.md` 登记“建立初步项目认知”任务。需要让当前认识指导设计或开发时，再建立 `docs/vcddd/systems/index.md`，将业务要求转化为系统、模块、领域操作和设计；单系统项目同样适用。不要在形成真实系统认识前为目录完整预建它。
- 若一次对话不能完成全部认识，在 `docs/vcddd/work/建立初步项目认知/index.md` 记录任务目标、完成条件、四阶段中的当前停留位置、已经形成的需求与业务全貌、目标线依赖顺序、模块覆盖、关键缺口和下一步。不要把阶段名当成任务名，也不要为每个阶段自动拆任务。
- 每项业务使用中文业务名建立独立目录，例如 `docs/vcddd/business/<中文业务名>/index.md`；其中 `index.md` 和详细业务文件只表达当前权威认识，潜在线索以文字保存在同业务目录的 `exploration.md`，未经确认不得进入业务图。
- 只有某条目标线开始产生足够多的独立内容时，才为它拆分业务视图；未探索目标线保留在地图和理解待办中即可。
- 将初步认识标明为当前解释、待确认推断或尚未探索，不要让文件落盘使低证据结论看起来已经定案。

首次档案的作用是让下一次对话知道项目为什么存在、准备实现哪些目标、业务与系统大致由什么构成、当前理解到了哪里以及下一步应继续什么，不是一次性完成全部需求、业务和系统设计。

### 恢复已有项目

适用于项目入口有效且本次请求延续一个或多个已有任务。

1. 读取 `docs/vcddd/index.md`。
2. 读取 `docs/vcddd/work/index.md` 的进行中任务及其目标摘要。
3. 用户只想了解项目且没有选择任务时，保持无任务状态；按问题读取业务总览、系统总览和相关当前认识，不读取全部任务详情。
4. 用户明确继续或开启任务后，按当前问题读取任务入口列出的认识：业务讨论读取相关 `docs/vcddd/business/`，系统设计或开发优先读取相关 `docs/vcddd/systems/`；不要默认同时读取两类全部材料。
5. 请求涉及多个任务时，分别读取并保持结论分离。
6. 仅在效力、理由、证据或跨任务关系不清时追溯决策、验证或历史任务。
7. 用任务目标、完成条件、有效理解、关键未知、进度和下一步校准任务行动。
8. 判断任务当前依赖需求发现、业务梳理、模块落地还是宏观检查的哪项成果；这只用于恢复位置，不把任务强制切成阶段。

无任务回复结束前，列出所有进行中任务的中文名称及目标或停留位置，并提醒用户可以明确继续某项任务或给出新的任务目标。没有进行中任务时只提示可以开启新任务。不要因为用户阅读、提问或了解项目就自动建立任务。

### 建立或修复入口

适用于用户已授权持续项目工作，但入口缺失、无法定位进行中任务或导航已经失效。

- 先从用户材料、当前仓库和可观察行为形成最小可用解释。
- 创建或修复 `docs/vcddd/index.md`，使其至少能指出项目与环境边界、`business/index.md`、`work/index.md`、存在时的 `systems/index.md`、最重要的开放问题和阅读方式。
- 让 `work/index.md` 分别列出进行中、已完成和已取消任务；允许存在多个进行中任务，不设置全局唯一当前任务。
- 为每个任务建立一个中文名称的稳定入口，说明任务目标、完成条件、范围与非目标、当前状态、涉及的业务与系统目录、有效选择、关键未知、验证状态、停留位置和下一步。
- 只创建已经有内容的文件。让小项目在入口内直接承载信息，待内容具有独立变化频率或按需读取价值时再拆分。

任务目录使用中文目标名称，例如 `docs/vcddd/work/<中文任务名>/index.md`。状态只写在任务入口与 `work/index.md` 中，不通过重命名目录表示。阅读导航应说明材料回答什么问题、何时读取、当前效力及其关联，而不只是列出路径。

### 维护进行中的任务

适用于纠正、新分支、方案选择、验证或实现发现改变了当前认识。

- 更新受影响的业务视图、系统视图或任务理解。
- 保存变化的来源、理由、证据强度、适用范围、限制与重新讨论条件。
- 同步检查相关图、文字、决策、验证和实现指导是否仍然一致。
- 分别更新受影响任务的开放问题、当前进度、下一步和阅读导航；一句反馈可以更新多个任务，但不得混写。
- 将问题留给用户前，先记录已经检查的证据、AI 的当前推荐、为何只有人能补充或决定，以及不同答案会改变什么。能够继续调查或自主判断的问题不得伪装成“等待用户确认”的阻塞项。
- 多个进行中任务影响同一业务、模块领域或设计时，先读取各任务最新状态并暴露冲突，不要让后写入的任务静默覆盖另一个任务依赖的认识。

### 建立、切换或完成任务

适用于出现新的独立目标、对话在多个任务间切换，或某项任务已经达到完成条件。

- 不把需求、设计、实现或验证自动拆成不同任务；只在目标能够独立完成、暂停、恢复或并行时建立新任务。
- 一个对话切换任务时明确任务名称；同一回复处理多个任务时分别更新和输出。
- 多个对话可以推进同一或不同任务。每次写入前重新读取任务入口和相关业务，避免依据陈旧状态继续。
- 任务达到预先说明或后来共同修正的完成条件时立即标记已完成；不要因为仍有范围外未知而推迟完成。
- 任务未达成但用户决定停止时标记已取消，不得冒充已完成。
- 完成时将持久业务认识同步到 `docs/vcddd/business/`，将持久系统责任、模块认知、领域模型与设计同步到 `docs/vcddd/systems/`，保存完成证据，并通知依赖该任务的其他进行中任务重新判断。
- 将只解释当时工作的讨论、验证细节和失败尝试留在任务历史中；当前业务认识只保存在 `business/`，当前系统与模块认识只保存在 `systems/`。

## 跳过与返回条件

- 用户只要求解释、审查或状态报告时，保持只读；不要擅自创建或修改认知空间。
- 用户尚未选择任务时保持只读了解；若新输入可能产生持久语义变化，说明影响并等待用户决定归入哪个任务或开启什么新任务。
- 入口和当前材料足以回答窄问题时，跳过无关历史、完整演化记录和其他任务。
- 当前材料的效力或来源不清、跨视图冲突、用户重开旧选择、或当前任务明确依赖历史证据时，再扩展读取。
- 存在多个进行中任务且用户意图无法唯一匹配时，先让用户选择；不要把某个任务永久设为全局当前任务。
- 无法从入口判断下一步应读什么时，先修复导航；不要让后续参与者猜测。
- 新证据改变任务目标、业务边界或当前选择时，返回任务入口和受影响业务视图，不要只改下游实现。

## 常见机械错误

- 首次建档从第一个页面或功能开始，没有先形成需求全景和宏观目标地图。
- 把用户认为完整的原型直接当成完整需求，没有区分可观察行为、推断和候选遗漏。
- 首次建档只创建当前局部需求，让后续 AI 看不见总业务目标和其他目标线。
- 把业务目标线列表直接变成开发目标、迭代计划或文件夹清单。
- 只保存目标线列表而不保存前提依赖和推荐理解顺序，使后续 AI 继续随意挑选线路。
- 把一个对话当作一个任务，或把一个任务绑定到创建它的对话。
- 用户只是进入项目了解情况时，自动选择最近任务、创建“项目了解任务”或开始修改业务档案。
- 无任务回复结束时不提示当前进行中的任务，使用户不知道可以从哪里继续。
- 只允许一个进行中任务，导致并行工作相互覆盖。
- 用英文缩写命名任务，或把 `进行中`、`已完成` 写进目录名并在状态变化时反复改路径。
- 把文档同步是否完成当作任务目标是否完成，或因仍可继续研究而拒绝结束已达成任务。
- 为符合目录外观预建空目录、空索引、空决策或空验证文件。
- 默认读取全部历史，让失效方案和历史任务干扰当前判断。
- 把代码、原型或运行现状当成期望业务规则。
- 无痕覆盖旧判断，或让候选方案与当前权威业务认识并列却不标效力。
- 用文件类型或 UML 类型拆分内容，另建脱离业务说明的图目录。
- 将操作日志、对话全文或原始推理写成“演化记录”。
- 把项目知识写进 Skill 的通用 `references/`。
- 系统目标已经明确后仍逐轮只抛一个小问题，没有先给出完整模块、领域操作与结果地图。
- 确认一个局部判断就宣布某个模块已经清楚，未覆盖其对象、完整命令与操作、内部状态影响、领域结果、事件和运行生命周期。
- 为每个模块或操作自动建立任务，导致一个完整系统目标被拆成无法看见全貌的微任务链。
- 把任务恢复点写成一串等待用户回答的问题，却没有保存 AI 已完成的调查、推荐模型和可继续推进的工作。
- 把阅读源码、已有测试或文档后的推断写成技术验证结论，没有建立并实际运行仓库根目录 `验证/` 下的专用项目。
- 把验证项目放进 `docs/vcddd/` 冒充第四类事实，或只保留实验源码而没有把命题、条件、结果和影响回写任务与相关基线。

## 三类事实空间与三层认知

项目认知区分三种描述对象。它们不是需求发现、业务梳理、模块落地和宏观检查四个阶段，也不能互相替代。可运行的技术验证项目位于认知空间之外，只作为结论证据，不构成第四类事实。

```text
<目标仓库>/
├── docs/vcddd/
│   ├── index.md                              # 唯一项目入口和阅读导航
│   ├── business/
│   │   ├── index.md                         # 整个业务的认知入口
│   │   └── <中文业务名>/index.md             # 内容足够独立时拆出的业务认识
│   ├── systems/
│   │   ├── index.md                         # 跨系统全景与系统选择导航
│   │   └── <中文系统名>/
│   │       ├── index.md                     # 该系统自己的认知入口
│   │       └── modules/<中文模块名>/
│   │           ├── 认知.md                  # 该模块自己的认知入口
│   │           ├── 领域.md                  # 模块领域行为
│   │           └── <按需设计文件>.md
│   └── work/                                # 进行中、已完成和已取消的工作记录
└── 验证/                                    # 按需创建的可运行技术验证项目
```

三类事实空间与三层认知不要混为同一分类：

- `business/`、`systems/`、`work/` 区分事实描述的是业务意义、开发基线还是工作变化；
- 业务、系统、模块区分参与者正在理解的范围和责任粒度；
- `work/` 不是一层业务认知；模块虽然位于 `systems/` 中，也必须拥有不能被系统入口替代的认知。

### 业务、系统与模块的认知责任

三个层级都必须建立自己的完整认识，但“完整”不等于混入所有下层细节。用每层唯一核心问题、禁止抢占内容和完成信号约束边界：

| 层级与入口 | 唯一核心问题 | 禁止抢占本层主叙述 | 完成信号 |
| --- | --- | --- | --- |
| 业务：`business/index.md`，必要时加相关业务入口 | 为什么要做、谁要通过哪些跨系统主线取得什么总体结果？ | 系统分工、模块、Domain、接口、数据和技术调用 | 不借助软件结构也能说清背景、参与者、目标、核心正常主线、直接结果与关键边界 |
| 系统：`systems/<中文系统名>/index.md` | 该系统在整个业务中独立承载什么，核心怎样产生结果？ | Domain 内部对象和命令细节、表、字段及调用机制 | 只读系统入口即可说明它为谁做什么、核心怎样运行、支撑闭环是什么以及由哪些模块承载 |
| 模块：`systems/<中文系统名>/modules/<中文模块名>/认知.md` | 该模块必须完成哪些领域事情并产生什么结果？ | 聚合内部细节、数据库、接口格式和跨域总流程 | 模块在系统中的目标、主要操作、语义输入输出、结果、边界与保证不再互相竞争 |

每个入口首先使未参与讨论的人回答“这是什么、要做什么”，随后回答“核心怎样完成、由什么辅助闭环”，最后提供“目前到哪里、接下来做什么”的导航。内容始终按“核心正常业务 → 支撑闭环 → 异常与恢复 → 技术细节”降序组织。它们是各自完整的认知，不是同一份全局材料的长、中、短摘要；风险和依赖可以改变分析顺序，不能改变认知入口的阅读顺序。

核心与辅助必须在每个层级重新判断。整个业务的辅助能力，可能正是某个系统被建立的核心能力；某个系统的辅助闭环，也可能是专门模块必须完整负责的核心领域。下层应通过自己拥有的责任重新表达怎样支撑父层目标，不能只抄父层业务描述；父层只保留当前范围所需的边界、主线、承载关系和链接，不能重复拥有下层操作、规则和设计。

正常阅读路径是：

```text
业务认知 → 系统全景 → 单个系统认知 → 单个模块认知 → 领域、设计与代码
```

从任务或窄问题进入时可以按需跳过无关分支，但进入某一层前应已经获得其父层上下文。`systems/index.md` 只是跨系统地图和选择入口，不能代替任何单个系统的认知；`领域.md` 负责可实现行为，也不能反过来代替模块全景。

模块之后继续按问题而不是按文件下钻：

| 认知对象 | 唯一核心问题 | 不属于它的内容 | 完成信号 |
| --- | --- | --- | --- |
| Domain | 当前领域怎样独立保持业务逻辑自洽并造成事实影响？ | 调用者、消费者、跨域编排、API、数据库和消息机制 | 对象、命令、规则、不变量、内部变化、结果、事件及失败恢复形成可实现因果链 |
| 业务编排 | 多个 Domain 为什么、按什么业务条件共同形成参与者结果？ | 事件、HTTP、事务和部署机制 | 独立领域结果能按业务理由闭合为完整路线 |
| 执行与技术设计 | 已确认的领域行为和业务编排怎样在现实约束下可靠运行？ | 新目标、新规则或尚未确认的领域语义 | 每项机制能追溯到领域事实、业务路线或明确运行约束 |

只有当前层核心问题已经能用当前层语言直接回答、关键分歧透明且得到共同接受，才能把下层草案提升为基线。若业务目标、系统存在目的、核心主线、模块边界或 Domain 所有权被纠正，依赖旧前提的下层内容全部标为 `待重新判断`，先返回受影响层重新建立基线；不得把旧图、领域模型、设计或数据结构继续当作既成事实。

### `business/`：需求目标与业务线事实

保存当前接受的业务意义：现实背景、参与者利益、总体目标、主要大目标、目标收益与关系，以及为了完成它们而形成的核心业务主线、各主线的支撑闭环、环境前提、判断与规则、业务流程和结果、隐藏规则及适用范围。业务线可以跨越任意系统和模块，不按照当前软件结构拆分。宏观需求与目标图、业务流程图、状态图和决策图与对应说明共同保存在这里。

不要填写系统分工、模块结构、接口调用、当前任务进度或尚未接受的方案。与某条业务长期相关但尚未确认的潜在线索，可以明确标为非权威文字放在该业务目录的 `exploration.md`；解决它的行动仍属于具体任务。

`business/index.md` 是整个业务认知的入口，不是系统和模块目录。它先表达最大的业务要做什么、为何存在、由哪些跨系统主线实现总体结果，再导航到值得独立读取的业务认识和承载系统；具体系统能力、模块分工与领域操作不在这里取得权威。

最终检查标准是：只读取 `business/`，人或 AI 先能说明现实背景、项目为什么存在、哪些参与者为了什么直接结果进入业务、核心业务主线是什么、当前还应认识或推进什么，再能按需下钻支撑闭环、规则和异常，而不需要先理解软件怎样拆分。

### `systems/`：系统开发事实

保存当前接受的开发基线：现实系统背景与建设边界、系统服务对象和核心业务概览、系统责任与非责任、模块认知，以及包含领域对象、命令、操作、规则、内部状态影响、领域结果和事件的领域模型；调用来源、事件订阅或直接调用等应用协作，以及接口、数据、交互、安全、并发、测试等设计按需保存在所属系统或模块。非简单系统和模块用已确认图作为结构、顺序、状态与因果关系的权威，文字补充图无法精确承载的规则和限制。系统是业务实际执行的承载者，但其材料按照自身模块和领域行为组织，不按照某条业务目标线建立功能副本。系统入口保留理解当前系统所需的核心业务主干，详细跨域业务流程仍以 `business/` 为权威。DDD 专有概念用于明确具体业务概念的语义角色，不代替项目自身的业务语言，也不作为必须填满的术语清单。

业务要求应被转化为各 Domain 自己的语义输入、命令、判断、内部对象变化、结果、事件、失败语义和保证，并保留来源链接。多个 Domain 为什么共同参与、先后或并行关系以及最终业务结果仍在 `business/`；事件订阅、直接调用和传输契约等执行协作保存在系统入口或所属模块设计；任何一层都不得把外部编排、消费者行为或存储方式塞回单个 Domain。

`systems/index.md` 只负责跨系统全景与选择导航；每个 `systems/<中文系统名>/index.md` 都要建立该系统自己的完整认知，不能只留下系统定义、模块列表或从业务入口裁剪的一段摘要。最终应做到首次阅读单个系统入口即可知道这个系统在整个业务中承载什么能力和目标、核心做哪些事情、由哪些辅助闭环和模块完成、当前到哪里以及下一步进入哪里；继续读取相关模块的 `认知.md`、`领域.md`、必要设计与实际代码就足以指导开发。只有语义来源或冲突不清时才需要返回业务手册。具体目录与内容要求见 [system-understanding.md](system-understanding.md)。

### `work/`：工作记录

保存实际工作记录：任务目标与完成条件、范围、状态、受影响的业务和系统、候选变化、备选与理由、验证命题及项目链接、实际结果与证据强度、已接受风险、冲突、当前停留位置和下一步。

任务入口只描述相对当前业务或系统基线的变化，不复制完整基线。候选规则、系统拆分、模块改责或领域设计方案在任务中存在，只能证明“项目正在考虑或验证它”；获得共同接受后，分别提升到 `business/` 或对应模块才成为当前基线。已经有依据的模块正文不因局部未知全部留在任务中：`认知.md` 持续保存模块全景和覆盖状态，`领域.md` 持续保存已经成立的对象、命令、操作、内部影响、结果和事件；跨域业务编排与应用协作分别维护，不塞入 Domain。

任务目标是完成或梳理完整系统时，任务入口应保存需求目标、业务目标线、系统、模块和实现的当前覆盖关系，四阶段中的停留位置，以及当前“系统 → 模块 → Domain 内部命令、操作、对象影响、结果和事件 → 业务层跨域编排 → 应用协作与必要技术投影”编排、模块依赖、各模块落地状态、宏观闭合缺口和整体完成条件。记录 `未展开`、`已有完整草案`、`已确认`、`待实现` 或 `待验证` 等真实状态即可，不把阶段变成不可返回的门禁。不要让恢复点只写“下一次讨论某个小问题”，使新的 AI 看不见剩余完整范围；也不要把只完成名词定义、对象关系或能力清单的模块记为已经完成。

模块地图成立后就建立或更新其 `认知.md` 和待确认全景图；每一组领域行为形成足够依据后立即写入 `领域.md` 及相应对象、操作、状态和事件图；设计选择形成后写入所属模块的按需设计文件。用户确认后将图标记为当前权威。即使整个模块或系统任务仍在进行，也同步在 `work/` 更新覆盖状态。尚未解决的备选、证据、AI 推荐和分歧保留在任务中，不得混入权威图，也不得让已完成部分只停留在任务或对话中。

最终检查标准是：新的 AI 读取任务入口及其直接链接后，能够知道目标、已经做到哪里、当前依据什么、还差什么以及下一步怎样继续，而不必恢复原始对话。

### 放置与同步规则

先根据内容实际回答的问题判断位置：

| 内容回答的问题 | 权威位置 |
| --- | --- |
| 为什么要做、为谁改变什么、有哪些目标与收益、什么不可接受 | `business/index.md` 与相关业务材料 |
| 参与者怎样通过业务路线、判断、规则和结果实现目标 | `business/` |
| 当前有哪些系统与模块，各自负责完成什么并提供什么保证 | `systems/index.md` 与系统入口 |
| 当前 Domain 有哪些对象、命令和查询；命令怎样判断、改变内部状态并产生领域结果与事件 | 对应模块的 `认知.md` 与 `领域.md` |
| 多个 Domain 为什么同时或先后参与、怎样形成总体业务结果 | 跨域业务路线在 `business/` |
| 事件触发还是直接调用、调用来源与消费者、数据库、接口或其他实现选择怎样落实领域行为 | 系统入口与对应模块下的应用协作、数据或其他按需设计文件 |
| 多个系统怎样共同完成业务目标 | 业务路线在 `business/`；高层依赖在系统入口；具体契约归提供方模块 |
| 这次准备改变什么、有哪些候选、证据、进度和下一步 | `work/` |
| 某项技术命题怎样被实际运行和复现 | 可运行项目在仓库根目录 `验证/<中文验证项目名>/`；其命题、运行条件与结果同时由相关任务记录 |
| 某项验证证明了什么 | 验证过程在 `work/`；被接受的业务或系统结论提升到对应基线 |

- 内容根据“它在描述什么对象”归类，而不是根据它来自需求讨论、设计还是代码分析归类。
- 一句话可能同时改变业务、系统和任务事实；分别更新受影响位置，不把整段内容复制三次。
- 业务规则的唯一权威位置是 `business/`；系统责任的唯一权威位置是 `systems/`；任务状态与未提升变化的唯一权威位置是 `work/`。
- 其他位置只保存必要摘要、差异和链接。发现重复结论时先确定权威归属，再删除或改为引用。
- `验证/` 保存可复现的实验源码、依赖、运行说明与必要证据；它不直接定义业务或系统事实。实验结论只有写入任务并在获得接受后提升到对应基线，才会改变当前认知。
- 已确认图在其表达范围内是权威；文字、设计或代码与图冲突时必须显式解决并同步更新，不能让文字静默覆盖图。
- 文件存在、目录齐全、链接有效、图可渲染、表格完整和术语出现只证明产物结构可读取，不证明当前层的核心问题已经被正确回答。语义基线必须由证据、层级完成信号和共同接受来判断。
- 数据设计从 Domain 已经成立的状态变化和系统实际查询、历史、审计、恢复与运行需要推导。表、字段、索引、约束和事务必须能链接回这些依据，但“是否持久化、怎样持久化”不写入 `领域.md`；数据库反向要求改变命令、聚合或规则时，先通过任务暴露冲突并重新判断，不能静默覆盖 Domain。
- 实现观察若与 `systems/` 冲突，先在任务中记录偏差并判断是实现缺陷、文档失效还是有意演化；不要直接用代码覆盖系统认识。
- 业务变化若改变系统必须提供的行为或保证，通过任务把变化翻译成模块操作、对象影响、事件和设计要求并同步到 `systems/`；不要要求开发者每次重新推导业务含义。
- 系统变化若改变参与者利益、业务判断或业务结果，必须同步回到 `business/`；只改变领域操作的实现、模块分工或技术方式时不要伪造业务变化。

## 必须保留的语义结果

保留足以让下一位参与者回答以下问题的内容：项目处在什么背景且边界在哪里；为哪些参与者实现什么总体目标、大目标和收益；有哪些跨系统与 Domain 的业务路线支撑它；每个 Domain 自己接收什么命令、怎样保持内部逻辑自洽并产生什么结果与事件；多个 Domain 为什么以及按什么条件共同参与业务；事件触发、直接调用、数据与接口等系统设计怎样落实但不污染领域；宏观组合是否已经端到端闭合；目前有哪些进行中、已完成或已取消任务；每个任务的目标与完成条件是什么；本次要继续哪个任务；它涉及哪些业务与系统认识；依据和适用范围是什么；仍有哪些分歧、风险和未知；当前做到哪里；下一步从哪里继续。
