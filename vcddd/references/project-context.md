# 项目上下文与目标管理

## 工作目的

使没有历史对话的人或 AI 从一个稳定入口恢复当前目标、有效认识、进度和下一步，同时避免把全部历史重新装入上下文。

将目标仓库中的 `docs/vcddd/` 作为项目专属共同认知空间。将 Skill 自身的 `references/` 视为跨项目行动规则；不要混写两者。

## 判断变量

先判断：

- `docs/vcddd/index.md` 是否存在，并能否指出当前目标和必要材料；
- 本次请求属于哪个目标，还是形成了结果或边界不同的新目标；
- 哪些内容是项目当前基线、目标内已接受变化、候选或分歧、历史或已替代内容；
- 当前问题涉及哪些业务关注点，是否需要理由、证据或旧目标才能判断；
- 文档、图、原型、代码和运行行为之间是否冲突；
- 本次变化是否会跨后续目标继续成立；
- 当前请求是否允许写入项目文件：持续构建、修改或迭代通常包含认知维护，单纯解释、审查或报告则保持只读。

## 选择行动

### 恢复已有项目

适用于项目入口有效且本次请求延续已有工作。

1. 读取 `docs/vcddd/index.md`。
2. 确定本次目标和关注点，读取该目标入口。
3. 读取与关注点直接相关的项目基线和目标变化。
4. 仅在效力、理由、证据或跨视图关系不清时追溯决策、验证或旧目标。
5. 用当前目标、有效理解、关键未知、进度和下一步校准本次行动。

### 建立或修复入口

适用于用户已授权持续项目工作，但入口缺失、无法定位当前目标或导航已经失效。

- 先从用户材料、当前仓库和可观察行为形成最小可用解释。
- 创建或修复 `docs/vcddd/index.md`，使其至少能指出系统与边界、当前目标、项目基线入口、最重要的开放问题、最近的语义变化和按关注点阅读的材料。
- 为当前目标建立一个可恢复入口，说明目标、动因、范围、期望结果、当前理解、有效选择、关键未知、验证状态、影响的业务视图、停留位置和下一步。
- 只创建已经有内容的文件。让小项目在入口内直接承载信息，待内容具有独立变化频率或按需读取价值时再拆分。

阅读导航应说明材料回答什么问题、何时读取、当前效力及其关联，而不只是列出路径。目标入口可以位于 `docs/vcddd/goals/<goal>/index.md`，但已有项目采用其他清晰路径时不要为目录一致性搬迁。

### 维护进行中的目标

适用于纠正、新分支、方案选择、验证或实现发现改变了当前认识。

- 更新受影响的业务视图或目标理解。
- 保存变化的来源、理由、证据强度、适用范围、限制与重新讨论条件。
- 同步检查相关图、文字、决策、验证和实现指导是否仍然一致。
- 更新入口中的开放问题、当前进度、下一步和阅读导航。

### 切换或完成目标

适用于期望结果或边界已经实质改变，或当前目标的实际结果已经完成。

- 不把需求、设计、实现或验证当作不同目标；它们可以在同一目标内反复发生。
- 新目标出现时，明确它与既有目标的关系；并行目标不得无提示混写。
- 目标完成时，将跨后续目标继续成立的业务行为、术语、规则、判断理由、约束和证据边界提升到项目当前基线。
- 将只解释当时方案的讨论、验证细节和失败尝试留在该目标中作为历史。
- 更新项目入口，使默认上下文只指向当前系统和新的当前目标。

## 跳过与返回条件

- 用户只要求解释、审查或状态报告时，保持只读；不要擅自创建或修改认知空间。
- 入口和当前材料足以回答窄问题时，跳过无关历史、完整演化记录和其他目标。
- 当前材料的效力或来源不清、跨视图冲突、用户重开旧选择、或当前目标明确依赖历史证据时，再扩展读取。
- 无法从入口判断下一步应读什么时，先修复导航；不要让后续参与者猜测。
- 新证据改变目标、边界或当前选择时，返回目标入口和受影响业务视图，不要只改下游实现。

## 常见机械错误

- 为符合目录外观预建空目录、空索引、空决策或空验证文件。
- 默认读取全部历史，让失效方案和旧目标干扰当前判断。
- 把代码、原型或运行现状当成期望业务规则。
- 无痕覆盖旧判断，或让候选方案与当前基线并列却不标效力。
- 用文件类型或 UML 类型拆分内容，另建脱离业务说明的图目录。
- 将操作日志、对话全文或原始推理写成“演化记录”。
- 把项目知识写进 Skill 的通用 `references/`。

## 必须保留的语义结果

保留足以让下一位参与者回答以下问题的内容：系统为什么存在且边界在哪里；当前目标要改变什么；现在采用什么业务理解与选择；依据和适用范围是什么；仍有哪些分歧、风险和未知；当前做到哪里；最近发生了什么有业务意义的变化；下一步从哪里继续。
