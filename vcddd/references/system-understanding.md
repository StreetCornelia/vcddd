## 工作目的

在业务全貌已经形成以后，将业务要求转化为系统自身完整、正确且可组合的模块行为、明确的 DDD 领域模型和可维护设计。让开发者只读取相关系统材料与实际代码，就能判断系统应该提供什么、怎样拆分模块、每个模块由谁完成哪些事情、接收和产生什么、哪些领域概念承担行为、必须保持哪些规则和保证，而不必重新加载整份业务手册。

`docs/vcddd/systems/` 保存项目当前接受的开发基线：系统边界、模块认知、领域模型、状态归属、依赖保证和相关设计。它不按某条业务目标线组织，也不是从代码目录自动生成的结构清单。系统模块仍需保留业务语义和来源，避免“与业务目标分开”被误解为“可以脱离业务正确性”。

## 先区分业务线、系统和模块

- **业务目标线**围绕一个业务子目标组织，回答为了完成总体业务目标，人和系统需要经历什么、遵守什么并得到什么结果。它可以跨越任意系统和模块。
- **系统**是承载业务实际执行的稳定责任边界。它按照自身应提供的能力、状态和保证被理解与演化，不以单条业务目标线作为边界。
- **模块**是系统内部为了完整完成一组相关领域事情而形成的稳定责任单元。它拥有行为、判断、状态和对外结果，是系统文档中指导开发的主要维护单位，但不应被写成独立业务目标。
- **能力**是模块能够稳定完成且可判断正确性的业务承诺，用来检查模块行为是否完整，不再作为与模块并列的固定目录类型。
- **跨系统关系**说明系统之间存在什么依赖和责任交接；具体接口和结果必须归属于提供它的模块，不能由一个没有所有者的“协作”目录承担。

“不以业务目标线组织”不等于系统完全不关心业务。业务线说明为什么需要某些行为；系统材料把这些行为归入拥有它们的模块，完整保存参与者、输入、判断、动作、状态变化、输出、影响和保证，使多个业务目标能够共同使用同一模块能力。

若所谓模块拥有独立能力边界、能够独立演化，并向其他系统提供稳定保证，应重新判断它是否实际是一个系统。若所谓系统只是代码包、页面区域或部署方便形成的单元，没有独立能力责任，则不要仅因技术形态将其提升为系统。

## 让系统与模块各自拥有认知

业务、系统和模块不是同一份认识的三个摘要长度。业务层说明整个业务怎样取得总体结果；系统层说明某个系统在该业务中独立承载什么能力与目标、怎样通过自己的核心业务线路和辅助闭环完成它；模块层说明某个模块在系统中必须完成哪些领域事情、操作和结果。三层分别维护：

```text
docs/vcddd/business/index.md
→ docs/vcddd/systems/index.md
→ docs/vcddd/systems/<中文系统名>/index.md
→ docs/vcddd/systems/<中文系统名>/modules/<中文模块名>/认知.md
→ 领域.md、按需设计与代码
```

`systems/index.md` 是跨系统地图，不是某个系统的认知。不能从业务全局直接跳到模块清单，也不能把单个系统入口写成“系统定义 + 责任边界 + 模块链接”。每个系统入口必须让新参与者不读模块细节也能知道：它在整个业务中的位置、服务谁、承载什么能力和目标、自己的核心线路怎样产生结果、有哪些辅助闭环、模块怎样承载、当前到哪里以及接下来进入哪个模块。每个模块 `认知.md` 则必须独立说明：它为何存在、承担哪些核心领域事情、这些事情怎样从触发走向结果、有哪些辅助操作和依赖、当前覆盖到哪里以及接下来进入哪个领域或设计。

核心和辅助是相对于当前层级的判断，不是业务层一次分类后永久继承。一项能力可能只是整个业务的支撑，却是某个系统被建立的核心原因；它也可能只是该系统的一段辅助闭环，却是某个专门模块必须完整负责的核心领域。下层从自己的责任重新表达上层目标怎样被实现，不能只裁剪或重复上层描述；上层保存边界、主线、承载关系和链接，不重复拥有下层操作、规则和设计。

### 按层级完成，不按内容堆积

进入系统之后仍必须一次只回答当前认知对象的核心问题。下层内容可以作为证据或链接存在，但不能替上层完成：

| 当前对象 | 唯一核心问题 | 禁止抢占本层主叙述 | 完成信号 |
| --- | --- | --- | --- |
| 系统 | 这个系统在整个业务中独立承载什么，核心怎样产生结果？ | Domain 对象、命令细节、表、字段和调用机制 | 首次阅读者只看入口即可说明服务对象、系统目的、正常主线、直接结果、支撑闭环和模块承载 |
| 模块 | 这个模块必须完成哪些领域事情并产生什么结果？ | 聚合内部细节、数据库、接口格式和跨域总流程 | 上层目标、主要操作、语义输入输出、结果、边界与保证已经明确 |
| Domain | 当前领域怎样独立保持业务逻辑自洽并造成事实影响？ | 调用者、消费者、跨域编排、API、数据库和消息机制 | 每条命令的规则、内部变化、结果、事件及拒绝恢复形成可实现因果链 |
| 业务编排 | 多个 Domain 为什么、按什么业务条件共同形成参与者结果？ | 事件总线、HTTP、事务和部署方式 | 各 Domain 的独立结果按业务理由闭合为完整路线 |
| 执行与技术设计 | 已确认的领域行为和业务编排怎样在现实约束下可靠运行？ | 新业务目标、新规则和未确认的领域语义 | 每项机制都能追溯到既有领域事实、业务路线或运行约束 |

“完成”只表示当前对象的核心问题在当前范围内已被证据支持、关键分歧透明并得到共同接受，不表示所有细节永久封闭。当前问题未能直接回答时，不因文件、图、术语、表或测试数量增加而下钻。AI 先综合证据给出本层完整草案和推荐判断，再让用户集中纠偏；不得把尚未形成的认识拆成连续问答交给用户设计。

## 分开阅读、分析与实现顺序

同一系统至少存在三种用途不同的顺序：

- **读者理解顺序**：现实环境与边界 → 系统为谁解决什么 → 核心业务主线与直接结果 → 支撑主线闭环的能力 → 模块承载 → 领域和技术细节；
- **分析顺序**：根据证据缺口、业务前提、依赖关系和可能推翻方案的风险决定先研究什么；
- **实现顺序**：根据模块依赖、技术前提、交付范围和验证成本决定先开发什么。

不要把三者合并。某个内部机制可能被所有模块依赖，是依赖核心；某个异常恢复可能最容易推翻方案，是风险核心；只有直接解释当前对象为什么存在、参与者或上层对象依赖它取得什么结果的线路，才是当前层级的价值核心。系统可能主要服务其他系统，模块也可能承载上层的支撑能力；不能因为它们不是整个业务的用户主线，就否定其在本层的核心地位。复杂度、依赖数量和风险仍不能自动把其他支撑机制提升为首次阅读的主角。

AI 可以先分析最危险或最基础的问题，任务记录也可以按实际发生顺序保存探索过程；写入 `business/`、系统入口和模块认知时必须重新按读者理解顺序组织。最终文档不是 AI 推理日志，也不是开发计划的线性展开。

## 先形成系统全局编排

系统目标、边界和主要输入输出已经明确时，AI 应主动综合现有材料形成一份可批判的系统编排，而不是从第一个想到的问题开始逐项询问。在提问模块、能力或规则之前，先检查用户提供的资源、当前文档、原型、代码、测试和可观察运行行为；能够找到或合理推断的内容由 AI 自己完成，并标明证据与推断边界。

初稿至少同时展示：

- 系统所在的现实环境、相关外部主体、建设边界和必要背景；
- 哪些参与者或上层对象会为了什么结果进入、使用、管理或依赖系统，以及一句话能够说明的系统存在目的；
- 当前范围内的核心业务主线、每条主线的正常入口、主要事情和直接结果；
- 为各条核心主线提供完整性、安全、变化和恢复保证的支撑闭环；
- 当前范围内全部候选模块，而不只是材料最丰富的模块；
- 每个模块承载哪些核心主线或支撑闭环、要完成哪些主要领域事情并产生什么结果；
- 系统对外能力由哪些模块共同承载；
- 模块之间的前提依赖、状态归属、责任交接和共同保证；
- 哪些内容已有依据，哪些是 AI 推断、待确认或尚未探索；
- 推荐讨论顺序及整个系统认识的完成条件。

非简单系统必须先让人通过一张核心业务全景图看见“哪些参与者为了什么进入系统、完成哪些主要事情、得到什么结果”，再用“核心业务主线 → 系统模块 → 模块主要领域操作与结果”承载图和依赖图表达完整编排；两种视角足够清楚时可以合并，不能用模块图替代业务主干。问题清单、功能待办、模块定义或若干待确认主题不能替代这些视图。先用参与者目的、核心主线、系统目标、所有对外输入输出、必须保持的不变量和现有材料做覆盖检查，避免遗漏没有被页面或文档直接命名的模块。初稿明确标为待确认，可以被整体纠正、重组甚至推翻；只有用户确认后的图才取得当前效力。

讨论顺序可以参考依赖和风险，但顺序只用于导航，不是一次只能讨论一项的门禁，也不改变文档中的读者顺序。信息足够时，一轮可以完成多个模块或整个系统的草案。不要为每个模块自动建立独立任务；只有模块本身形成了可单独完成、暂停或并行的工作目标时才拆任务。

## 完整讨论一个模块

进入某个模块后，目标不是给它下定义，而是得到足以指导开发的模块领域行为。统一语言若只统一了名词，仍无法保证软件做出相同的判断和结果；还必须统一领域中的动作、规则与后果。先说明它承载哪些核心业务主线或支撑闭环、参与者为什么会触发它、实际要完成哪些事情并得到什么结果，再把每件事情从发生条件推导到可观察结果；先给出完整草案、推荐判断和必要备选，再让用户集中纠偏。不要用空白框架或问题清单要求用户从头设计，也不要让内部状态、事件或恢复机制出现在模块存在理由之前。

能力名称只能用于检查覆盖，不能作为模块内容本身。每项能力都必须落实为一个或多个领域操作及其事件因果链；一个模块通常同时包含多个相互关联的闭环。可用下面的关系检查是否真正说明了行为，但不要把它强制写成固定表单：

```text
进入领域边界的命令 → 领域操作与判断 → 内部对象、属性或事实变化 → 领域结果与事件
                         ↘ 拒绝 / 失败 / 取消 / 重复 / 恢复
```

模块讨论至少覆盖适用的以下内容：

- 存在目的、责任、明确非责任及它在系统中的位置；
- 它需要完成的全部领域事情，以及这些事情共同形成的完整能力，而不是当前正在谈的一项功能；
- 每件事情以什么领域命令或外部事实进入边界，接收什么语义输入，依赖什么领域前提；
- 为完成事情需要执行哪些领域动作、作出哪些判断、遵守哪些规则，先后或协作关系是什么；
- 每条正常路线怎样结束，改变了哪些对象、属性、关系、状态或事实，产生什么输出和完成事件；
- 当前领域会产生哪些准确陈述自身影响的结果与事件，但不在这里指定消费者；
- 配置与准备、启用与正常运行、规则或状态变化；
- 失败、超时、降级、取消、重复、中断和恢复；
- 停用、替换、迁移或退出对新执行、在途执行和历史事实的影响；
- 相关跨域业务路线与应用协作的链接、当前领域的可观察结果、验证证据与剩余未知。

这里的“模块生命周期”是对模块所承载能力和状态在完整运行过程中的覆盖检查，不要求每个模块生造一套状态机，也不是固定表单。只展开会改变能力正确性、系统保证或开发实现的内容。

不要把上述内容拆成连续多轮确认题。AI 应基于证据先写出完整草案，集中列出真正会改变模块结构或语义的分歧。只有检索现有证据后仍缺少仅由人掌握的信息，或存在必须由人承担后果的选择，才提出阻塞性问题；其他不确定内容以明确假设继续建模。

一个模块只有在已确认权威图中覆盖领域对象、完整命令与操作、对象与状态影响、领域结果与事件、异常路线、依赖和对外保证，文字补足图无法表达的精确规则，剩余未知透明且不再阻塞开发时，才算形成完整模块认识。只列出职责、能力名称、对象关系或若干术语，即使这些定义都正确，也不算完成；只有文字没有完成图上确认，也不算完成。单个 Domain 的完成只表示其内部业务逻辑自洽；整个系统还必须在业务和系统层检查多个 Domain 怎样共同闭合目标。完成一个模块不等于完成整个系统。

## 先确认领域自治，再设计协作与数据

业务全貌和模块职责足以提出行为要求后，先建立 Domain 的对象与领域命令模型，不先建立数据库、接口或跨域调用模型。这里的命令是进入领域边界、请求它尝试完成某项业务行为的意图；“命令内部流程”实际指命令被接收后在当前 Domain 内形成的领域处理因果链，不能把 Command 数据结构本身误当成领域行为。

先给出当前模块尽可能完整的命令地图，并把每个会改变领域事实的命令展开为：

```text
进入领域边界的命令及语义输入
→ 领域前提、策略与不变量判断
→ 聚合、实体、值对象或领域服务执行操作
→ 当前 Domain 内部的状态、关系或业务事实变化
→ 当前 Domain 产生的领域结果与领域事件
  ↘ 任一步发生拒绝、失败、取消、重复或恢复时，仍保持内部自洽
```

Domain 确认的是自己怎样忠实执行业务并造成影响，不是技术调用栈，也不是整个业务怎样组合多个领域。它不关心命令来自 API、页面、定时任务还是其他 Domain，不维护谁会消费自己的结果，也不规定外部随后必须做什么。生产领域只定义“本领域已经发生了什么”；消费领域只在自己的模型中定义某个外部事实进入边界后怎样转化为本领域命令和内部变化。

跨 Domain 的宏观编排由业务层维护：哪些业务情境会同时或先后需要几个领域、某个领域结果满足什么条件后继续哪条路线、最终怎样形成参与者结果。系统层再决定如何执行该编排：可以通过领域事件自动触发，也可以在没有事件系统时由当前调用链直接调用下一个领域；两种方式都只是协作与实现选择。即使调用代码物理上紧邻或位于前一个模块，语义上“接下来调用谁”仍属于执行编排，不属于前一个 Domain 的内部规则。不能让前一个 Domain 接管后一个 Domain 的逻辑，也不能把一条跨域业务路线塞进某个 Domain。

Domain 同样不考虑存储。它只定义当前业务对象、状态、事实和变化的含义；数据设计从外部观察领域命令会读取和改变什么，再结合业务查询、历史、审计、恢复、并发和运行条件，决定哪些内容需要保存以及怎样保存。每项表、字段、关联、索引、约束和事务必须能追溯到领域状态变化或系统实际需要，但这种映射属于设计文件，不回写成 Domain 的“持久化语义”。

AI 应先从业务材料、原型、代码和运行证据形成完整领域命令草案、内部因果图和推荐判断，再由用户确认真实业务语义与隐性规则；不要让用户逐条替 AI 设计命令，也不要用尚未确认的表结构、接口或跨域编排诱导用户接受一个领域模型。

## 从模块下推细节并及时写回

使用以下认知层次保持讨论焦点：

```text
系统全局：目标、边界、模块地图、依赖和整体保证
→ 模块认知：位置、参与者、完整操作地图、输入输出和结果
→ 模块领域：对象、命令、操作、规则、内部状态影响、领域结果与事件
→ 业务编排：多个 Domain 为什么先后、并行或按条件参与
→ 系统执行协作：通过事件触发、直接调用或其他方式落实业务编排
→ 模块设计：从领域事实和协作需要推导接口、数据、交互、并发、安全、验证等实现投影
```

全局已经确认后，优先沿既定模块地图推进，不因每个细节重新发起全局确认。AI 先基于证据说明当前模块的完整逻辑，再深入可能影响正确性的细节。细节中出现冲突时，先判断能否通过已有材料或验证解决；确实存在多种合理解释、人的隐性规则或需要人承担后果的选择时，集中给出差异、推荐和影响，再询问用户。

在用户已授权持续维护项目认知时，按内容形成速度持续写回，而不是等待整个模块完成：

1. 模块边界和完整操作地图成立后，建立或更新 `认知.md`；
2. 对象、命令处理流程、内部状态影响、领域结果和事件形成依据后，更新 `领域.md`；
3. 跨域业务顺序变化时更新 `business/`，事件触发或直接调用等应用协作变化时更新系统入口或所属模块设计；
4. 只有领域命令模型足以提供依据后，才更新该模块下的数据、接口或其他按需设计文件；
5. 将尚未解决的候选、分歧和验证需要保留在 `work/`，并在 `认知.md` 标明受影响范围；
6. 更新系统入口和任务入口中的模块覆盖、因果链缺口、受影响材料与下一步；
7. 检查该模块结论是否改变全局边界或其他模块，必要时返回对应层次修正。

每次写回同时更新受影响的权威图和补充文字。不要等待整个系统全部讨论完才一次性回忆并落盘，也不要把仍缺关键命令、内部状态变化、领域结果、事件或图上确认的模块标记为完整。模块目录中的已成立内容是当前系统基线，讨论流水和未接受方案仍属于工作记录。

## 用 DDD 专有概念固化通用语言

`systems/` 可以并且应该在语义已经出现时使用明确的 DDD 专有概念，使人、AI、文档、代码和测试能够引用同一个领域模型。领域不是一组静态对象：它由自身拥有的领域对象与状态、能够完成的领域操作、约束操作的规则与不变量、操作造成的内部事实变化，以及操作完成后产生的领域结果与事件共同构成。其他领域是否以及怎样响应，不属于当前领域模型。

```text
领域对象与状态 + 领域命令与操作 + 规则与不变量 + 内部事实变化 + 领域结果与事件
```

DDD 专有概念是用于说明这些语义角色的建模元语言，不是项目的通用语言本身。项目特有的业务名称、动作、规则和结果才构成通用语言；将具体业务概念标明为实体、值对象、聚合、操作或事件，是进一步确定它在领域中的身份、行为、边界和影响。

按实际语义选用以下概念：

| DDD 概念 | 在系统认识中确定什么 |
| --- | --- |
| 领域、子域 | 当前需要维持的对象、命令、操作、规则、内部事实变化、结果与事件所形成的知识和行为范围；不要把业务目标线、跨域编排、系统或目录直接当作子域 |
| 通用语言 | 在特定语境内共同使用的业务名称、动作、判断和结果；必须贯穿文档、代码与测试 |
| 限界上下文 | 某套通用语言和领域模型保持一致、具有明确效力的边界；它不天然等于一个系统或模块 |
| 上下文映射、防腐层 | 不同限界上下文之间如何关联、翻译和隔离语义，避免外部模型污染本地模型 |
| 实体 | 由业务身份连续性决定、会经历生命周期变化的领域对象 |
| 值对象 | 由整体取值和约束定义、没有独立业务身份的领域概念 |
| 聚合、聚合根 | 一组对象共同保持业务不变量的一致性边界，以及唯一允许外部操作该边界的入口 |
| 命令 | 请求系统尝试执行某项领域行为的意图；它可能被接受或拒绝，不代表事实已经发生 |
| 领域操作 | 领域为了完成某件事情而执行的行为；它依据规则读取或改变领域对象、属性和事实，并形成明确结果 |
| 领域事件 | 当前领域内某项操作已经完成并造成了值得关注的业务事实；它只陈述本领域发生了什么，不定义消费者及其后续行为 |
| 领域服务 | 真实领域行为无法自然归属于某个实体或值对象时，承载该行为的领域概念 |
| 策略、规格 | 可独立命名、组合、解释和验证的业务选择规则或判断条件 |
| 事件处理器、流程管理器 | 在消费方领域或应用协作中把外部事件转化为本领域命令，或协调跨聚合流程；不得把其他领域的内部行为并入生产领域 |
| 仓储、工厂 | 分别表达聚合的领域集合边界和复杂领域对象的有效创建；不是通用 DAO 或构造函数的改名 |
| 应用服务 | 编排一次系统用例、权限和事务等应用职责；不得把本应属于领域模型的判断藏在其中 |

模块、领域或限界上下文不存在默认一一对应关系。模块是系统内持续维护和实现一组领域行为的责任单元；限界上下文是通用语言和模型的语义边界；领域或子域是需要理解和解决的知识与行为范围。一个限界上下文可以由多个模块承载；若一个模块已经拥有独立语言、决策主权和演化节奏，应重新判断它是否构成新的限界上下文或系统。

把领域操作与应用用例明确分开：

```text
命令或外部事件
→ 应用用例完成调用适配、权限、事务和所需外部协调
→ 调用聚合、实体或领域服务完成领域操作
→ 领域操作依据规则改变对象与事实并形成领域事件
→ 应用与基础设施按设计保存结果、交付事件或直接调用其他领域
```

应用用例可以按照已经成立的业务路线协调多个领域操作，但不能自行创造领域规则；领域操作必须能够完全用当前领域的通用语言解释，不依赖 HTTP、消息队列、数据库、调用者身份或外部消费者等词汇。跨领域的先后和条件以业务材料为依据，事件或直接调用只是应用实现方式。

这些概念按需出现，不要求每个模块凑齐全部名词。识别时以真实语义为依据：是否存在业务身份、整体一致性、已发生事实、可独立表达的规则或语义翻译需要。每个改变领域事实的操作都应明确其状态影响和完成事件；纯查询只返回观察结果时，不要为了形式生造事件。不要把每个数据结构称为实体、每个 API 请求称为命令、每条技术消息称为领域事件，或用一个巨大聚合包住整个模块。

模块材料应把 DDD 概念放回当前领域内部的行为和事件因果链中，而不是只建立一份术语表。读者需要看见：进入领域边界的业务意图是什么；哪些实体、值对象、聚合、领域服务或策略完成操作；操作改变哪些内部对象、属性和事实；哪些不变量必须同时保持；完成后产生什么领域结果和事件。领域模型图和命令—操作—事件图可以作为主视图，但必须由规则、边界和失败语义补足图中无法表达的内容。谁发出命令、谁消费事件以及多个领域怎样形成完整业务结果，分别由业务路线和应用协作材料表达。

同一业务含义在一个限界上下文内只使用一个名称；同一词语在不同上下文中含义不同时，显式标明上下文，不用全局词汇表强行合并。已经接受的 DDD 角色、聚合边界和上下文关系属于 `systems/` 当前基线；仍在比较的命名、职责或边界候选保存在 `work/`，获得依据和共同接受后再提升。

## 识别并提升子域

领域模型允许随着理解深入逐步分解。一个概念最初可能只被当作父领域中的对象；当证据显示它已经形成可独立理解和维护的问题空间时，AI 应主动提出将它提升为父领域下的子域，而不是继续把所有行为塞进一个巨大领域文件或聚合。

以下是子域信号，需要结合出现，不能把任何单项当成机械门槛：

- 解决一组能够独立说明的业务问题，并产生可判断的结果；
- 拥有稳定业务身份、自己的通用语言和完整生命周期；
- 包含多项相互关联的领域操作，而不只是被父对象创建、修改或删除；
- 独立维护规则、不变量、状态、事件、失败和恢复语义；
- 被父领域的多个行为、其他子域或其他模块通过稳定语义引用；
- 具有不同于父领域其他部分的变化原因、证据来源或读写节奏；
- 单独阅读、实现和验证它，能够得到完整行为闭环。

拥有身份或生命周期并不自动成为子域。若概念完全受一个父聚合控制、没有独立业务问题和语言，它仍可能只是实体、值对象或聚合内部对象；若它进一步拥有独立决策主权、对外契约、语言边界和演化节奏，则重新判断是否应提升为模块、限界上下文或系统，而不是无限嵌套子域。

提升子域时必须完成语义迁移，而不是只创建目录：

1. 在当前任务中记录提升依据、受影响行为和竞争性解释；
2. 明确父领域继续拥有的责任、子域独占的对象、操作、状态、事件和不变量；
3. 重新确定聚合根、引用和一致性边界；同一对象不能在父领域中是聚合内部实体，同时又在子域中作为独立聚合根取得第二份权威；
4. 将详细语义移入子域，父级 `领域.md` 只保留模块领域全景、子域关系、跨子域操作与共同保证，并通过链接引用；
5. 同步更新父子领域图、状态所有权、领域事件、事务与恢复边界，以及受影响的设计、代码和测试映射；
6. 检查原本由单一聚合保证的整体性在拆分后由谁协调、怎样失败和恢复，不能用目录拆分静默改变业务保证。

子域是语义分解，不等于代码包，也不要求一个子域只含一个聚合。不要为了缩短文件、目录整齐或看到一个复杂类就生造子域。

## 用图完成理解与最终确认

图是系统和领域认识的主要表达与确认工具，不是完成文字后的装饰。只要结构、流程、状态或事件关系具有两个以上分支、参与者或依赖，AI 就应主动选择合适的图形成完整草案，让用户直接在全貌上纠偏；不要先写长篇说明，再用一张内容更少的图概括。

非简单系统和模块至少保持以下权威视图：

- `systems/index.md` 或系统入口中的系统—模块全景图：表达系统、模块、主要操作结果、依赖和责任交接；
- `认知.md` 中的模块行为全景图：表达谁触发哪些操作、主要输入输出、结果和外部影响，并覆盖完整模块范围；
- `领域.md` 中的领域对象与聚合关系图、命令操作与内部状态影响图、领域结果与事件图。内容能够清楚合并时可以合成一张，复杂时拆成多个相互引用的切面，但必须保留一张能够导航当前 Domain 的总图；
- 设计文件中按需使用时序图、组件图、部署图、数据关系图或其他视图，说明设计怎样落实领域行为。

根据要确认的问题选择表达，而不是固定一种图：对象与聚合关系适合领域模型图或类图；生命周期适合状态图；领域内部命令与操作适合流程图；判断与分支适合决策图；当前 Domain 的状态变化与领域事件适合因果图。跨 Domain 的业务先后与并行由业务流程图表达，事件触发或直接调用等执行顺序由系统设计中的时序图或泳道图表达，不得混入单个 Domain 权威图。图中使用当前层级的通用语言，不用代码类名或基础设施名替代领域含义，除非该图本身就是实现设计。

图的权威性遵守以下规则：

1. 讨论中的图明确标为“待确认草图”，允许包含有依据的推断和待纠偏关系；
2. 用户确认后，去除草图标记并将图更新为当前权威；图所表达的结构、顺序、状态和因果关系以这张已确认图为准；
3. 未确认、存在分歧或只用于探索的内容不得混入权威图，在邻近文字中标明并链接对应任务；
4. 文字只补充图无法精确承载的定义、规则、前提、数值、失败语义、证据和限制，不重复维护另一套关系；
5. 图、文字、领域文件、设计和代码之间出现冲突时，立即作为任务问题暴露，先确认真实语义再同步修正，不以“后写者”为准；
6. 模块操作、状态、事件或依赖发生变化时，同一任务内同步更新受影响图；没有更新权威图就不能宣布认知或设计同步完成。

图与它解释的认知、领域或设计保存在同一文件或紧邻位置，优先使用可审查和版本化的 Mermaid 等文本图。不要建立按图类型分类的独立目录，也不要让读者离开正文才能知道图的效力和上下文。

## 系统事实空间

需要说明当前软件怎样承载业务并形成可开发基线时，建立：

```text
docs/vcddd/systems/
├── index.md
└── <中文系统名>/
    ├── index.md
    └── modules/
        └── <中文模块名>/
            ├── 认知.md
            ├── 领域.md
            ├── 子域/
            │   └── <中文子域名>/
            │       ├── 认知.md
            │       ├── 领域.md
            │       └── <按需设计文件>.md
            └── <按需设计文件>.md
```

只创建已有真实内容且值得独立读取的目录和文件。`认知.md` 与 `领域.md` 是非简单模块的两个稳定阅读入口；只有识别出真实子域时才创建固定中文目录 `子域/`，不得使用 `domains/`、`subdomains/` 或中英文混写。其他设计文件按真实需要增加，不预建空的子域、接口、数据、交互或测试文档。单系统小项目同样需要系统认识；系统没有值得独立拆分的内部模块时，可以直接在系统目录维护 `认知.md` 与 `领域.md`，不要为了目录整齐生造模块。

系统和模块使用中文通用语言命名。`systems` 与 `modules` 是稳定导航接口，不代表代码包命名要求。不要固定建立 `capabilities/` 或 `collaboration/`；能力属于模块，跨系统关系与接口必须有明确所有者。

## 系统全景的内容责任

`docs/vcddd/systems/index.md` 应足以回答：

- 当前软件整体由哪些系统构成，各系统为什么需要独立存在；
- 每个系统的核心责任和明确非责任是什么；
- 每个系统通过哪些模块完成哪些主要事情并产生什么结果；
- 存在哪些限界上下文及其语义边界，它们与系统边界是否一致；
- 系统之间存在哪些必要依赖、责任交接和对外保证；
- 关键输入由哪个系统的哪个模块接收，关键输出与影响由谁产生并由谁使用；
- 当前系统划分依据什么成立，存在哪些已知限制或不一致。

适合时使用核心业务全景图和“核心业务主线 × 系统 × 模块 × 主要结果”映射。项目级系统全景先说明现实解决方案中有哪些系统、各自为哪些核心业务结果承担什么，再表达责任边界、模块供给、依赖和保证。它不复制业务手册的完整流程和规则，但必须保留足以让首次阅读者理解整个解决方案主要做什么的业务主干；只有系统和模块名称而没有参与者目的与直接结果，不算全景。

## 单个系统的内容责任

`docs/vcddd/systems/<中文系统名>/index.md` 保存该系统当前接受的结构事实：

- 它处在什么现实环境中，哪些内容属于建设对象，哪些只是背景或外部前提；
- 哪些参与者或上层对象会为了什么结果进入、使用、管理或依赖它，它在整体解决方案中为什么需要存在；
- 它承载哪些核心业务主线，各条主线从什么主要情境开始、完成什么事情并得到什么正常结果；
- 哪些能力只是让核心主线可配置、可改变、安全、可追溯或可恢复的支撑闭环；
- 它承担什么稳定责任，哪些事情明确不归它负责；
- 它有哪些模块，每个模块承载哪些核心主线或支撑闭环、完成哪些主要事情并产生什么结果；
- 它拥有或作出哪些系统级判断，维护哪些对外有意义的状态；
- 它向其他系统提供什么能力和可依赖保证；
- 它依赖其他系统或环境提供什么最低条件；
- 它与其他系统发生哪些主要责任交接；
- 它承载哪些限界上下文、通用语言和关键领域模型；
- 模块之间怎样依赖、交接状态和共同完成系统结果；
- 开发这些模块必须遵守哪些不变量、失败语义和可观察结果；
- 当前实现与已接受系统认识是否存在已知偏差。

系统入口不是完整业务手册、模块正文摘要堆积或分析过程记录。它必须建立该系统自己的认知：先用现实背景说明它在整个业务中的位置，再说明它独立承载的能力与目标、自己的核心业务线路及结果、辅助闭环、模块承载和系统级关系。详细跨系统业务流程仍链接到 `business/`，具体领域行为和设计下沉到拥有它的模块目录。不要为避免重复而删除使系统目标成立的业务语义，也不要把业务入口裁剪复制成所谓系统认识。

入口还要标明当前认知与建设覆盖、会改变系统判断的关键未知，以及下一步应认识、设计或开发哪个模块。首次阅读者只看这里，就应能回答“这个系统是什么、在整个业务中要做什么、核心怎样运作、接下来去哪里”，而不是必须先翻模块文件猜出系统目的。

## 模块目录的内容责任

`docs/vcddd/systems/<中文系统名>/modules/<中文模块名>/` 是持续维护该模块开发认知的唯一位置。目录中的材料共同回答：这个模块为什么存在、能完成哪些事情、每件事情怎样完成、由哪些领域概念保证正确，以及当前选择怎样投影为实际软件设计。

不要把一个模块拆成若干只包含能力名称的摘要文件。能力用于检查模块要完成的事情是否覆盖完整；真正的维护单位仍是模块及其行为。只有某项设计内容具有独立阅读价值和变化频率时，才在该模块下增加文件。

## 模块认知的内容责任

`认知.md` 是进入模块时首先读取的入口和当前全景，不是空泛的“模块是什么”。它至少应使读者知道：

- 模块在上层系统中承载什么目标；哪些领域事情、操作和结果是这个模块自己的核心，哪些只是使其闭环的辅助能力；
- 模块在系统中的位置、存在原因、责任和明确非责任；
- 谁使用、触发、管理或受到它影响，谁对关键决定和结果负责；
- 模块能完成哪些主要事情，每件事情从什么情境开始并产生什么结果；
- 模块整体接收什么语义输入、输出什么结果、改变什么状态或事实、产生什么外部影响；
- 它依赖哪些模块、系统或环境，对外提供什么保证；
- 当前领域是否包含独立子域，各子域负责什么，父领域怎样协调它们；
- 各项领域行为当前是已成立、待验证、存在分歧还是尚未覆盖；
- 应继续读取 `领域.md` 的哪些部分，以及有哪些按需设计、实现、测试和任务入口。

非简单模块的 `认知.md` 必须先让读者看见它在系统中为何存在、自己的核心领域事情、参与者怎样触发这些事情并得到什么结果，再展开辅助能力、内部状态、异常和恢复。包含一张经用户确认的模块行为全景图，以它作为参与者、操作范围、输入输出、结果和外部影响的权威；再用文字补充边界、保证、覆盖状态、关键未知和下一步阅读或工作入口。`认知.md` 负责该模块自己的导航和全貌，不复制 `领域.md` 的完整规则或设计文件的实现细节。

## 模块领域的内容责任

`领域.md` 保存足以指导实现和验证的当前 Domain：内部对象、属性、状态、关系和不变量，进入边界的命令或业务事实，领域需要完成的全部操作，操作对自身对象和事实造成的影响，以及完成后产生的领域结果与事件。对象模型、命令与操作模型、内部状态变化和事件因果链缺一不可。它不描述调用来源、外部消费者、跨域业务路线或存储方式。存在子域时，父级 `领域.md` 还必须说明子域地图、语义所有权、父领域内部的跨子域操作和共同保证，但不复制子域内部完整模型。

非简单模块必须同时形成相互引用并经用户确认的权威视图：领域对象与聚合关系、领域命令与操作流程、操作与内部状态影响、领域结果与事件。它们可以合并在一张适合阅读的图中，也可以按复杂度拆开；图不是装饰，必须让人直接看见“领域接收什么业务意图、由哪些领域对象判断和执行、改变什么、产生什么结果与事件”。文字再补充图无法准确表达的规则和失败语义。

每项领域操作至少需要表达以下语义；表达形式可按内容选择文字、表格、流程图、状态图或时序图，不要求机械填表：

- **领域意图与操作**：进入领域边界的业务意图是什么，当前 Domain 要完成什么事情；
- **触发与输入**：由什么命令或外部业务事实开始；输入包含什么业务信息和领域约束，不绑定 API、页面或具体调用方；
- **前提与规则**：进入前必须成立什么；由哪个聚合、实体、领域服务或策略作出哪些判断；
- **对象与状态影响**：操作读取哪些领域对象和属性，按什么业务顺序修改哪些聚合、状态、关系或事实，必须同时保持哪些不变量；
- **结果与完成事件**：当前领域形成什么业务结果；操作成功完成后产生哪些领域事件，事件准确声明本领域什么已经发生；
- **失败与恢复**：拒绝、失败、超时、取消、重复、并发冲突、中断和恢复分别意味着什么；
- **可观察完成**：什么事实证明操作成功、失败或尚未确定，测试应观察什么结果；
- **来源与限制**：它承接哪些业务要求，哪些细节仍待验证或决策。

领域事件是当前 Domain 已经发生的业务事实，不等于消息队列中的一条消息。生产模块的 `领域.md` 只定义事件名称、发生条件和事实内容，不维护订阅者清单或消费者行为；消费模块只定义某个外部事实进入自身边界后形成的命令、内部变化和新事件。生产与消费为什么在一条业务路线中相连，由 `business/` 说明；事件怎样发布、投递、重试和去重，或者是否改用直接调用，属于应用协作和后续设计文件。

这里的输入、结果和事件首先是领域语义，不必立即等同于 JSON、HTTP 或消息 Schema；但不能以“尚未设计接口”为理由省略。只有先明确领域输入、操作、内部对象变化、结果和事件，后续调用协作、接口、数据和代码设计才有确定依据。

只列通用语言名词、职责、能力名称、对象关系或 DDD 类型，不说明命令怎样在当前领域内被处理、操作怎样改变内部对象并产生什么结果与事件，不算领域模型完成。开发者读完 `领域.md` 后，应能够实现并验证当前领域内部逻辑；若要理解外部为什么调用它、谁消费事件、怎样持久化或多个领域怎样完成总体业务，应继续读取业务路线和相关设计，而不是要求 `领域.md` 同时承担这些责任。

## 子域目录与内容责任

`docs/vcddd/systems/<中文系统名>/modules/<中文模块名>/子域/<中文子域名>/` 是该子域当前语义的唯一维护位置。子域使用与模块相同的两个稳定入口：

- `认知.md` 说明它为什么从父领域独立出来、解决什么问题、完成哪些事情、接收和产生什么、自己的边界，以及怎样与父领域和其他子域协作；
- `领域.md` 保存其对象与聚合、领域命令与操作、规则与不变量、内部状态影响、领域结果与事件、失败恢复和可观察完成；
- 接口、数据、并发、验证或实现映射等只服务该子域的设计，放在同一子域目录；真正跨多个子域的模块级设计仍放在模块根目录。

父级 `认知.md` 和 `领域.md` 必须把子域显示在模块全景图与领域总图中，并链接到子域入口。父级只说明它为何存在、承担什么语义、与其他部分怎样交接；子域内部对象、状态机、操作表和事件规则只在子域文档取得权威。发现父子文件仍同时定义同一聚合、状态或操作时，先解决所有权冲突，不能靠“详略不同”让两份定义并存。

进入模块时仍先读取模块 `认知.md`；只有当前问题涉及某个子域时才读取该子域 `认知.md`，需要实现或改变其领域行为时再读取子域 `领域.md` 和相关设计。不要因为存在 `子域/` 就默认加载全部子域。

## 模块设计文件的内容责任

接口、数据、应用协作、交互、并发与事务、安全、部署、可观察性、验证或实现映射等设计，都放在其服务的模块目录中。可按真实问题命名为 `接口设计.md`、`数据设计.md`、`应用协作设计.md`、`交互设计.md`、`并发与事务.md`、`安全设计.md`、`验证设计.md` 或其他中文文件；这些只是可能的文件，不是必须创建的清单。内容很少时可以保存在 `认知.md` 的设计导航中，但不得把存储、API、事件设施或跨域调用写进 `领域.md`。

每份设计必须指出它落实 `领域.md` 中哪些操作、规则、不变量、输入输出和影响，并说明当前实现、代码、测试与领域模型的对应关系。设计可以改变技术方式，不能静默改变领域语义；若设计发现领域模型无法实现或存在矛盾，先回到任务暴露问题，再修正 `领域.md`，不能让后写的设计文件自行覆盖领域事实。

数据设计还必须逐项说明它承接哪些领域命令造成的状态变化，以及哪些业务查询、历史、审计、恢复、并发或运行要求需要保存这些内容。不要从对象名直接生成一对象一表，也不要因框架惯例预建通用字段、扩展关系或未来表；表、字段、索引、约束和事务边界无法追溯到领域事实或明确系统需要时，不应进入当前基线。该追溯关系由数据设计维护，不反向成为 Domain 的存储职责。

## 完成模块技术投影与落地

模块职责和 Domain 设计成立后，继续把领域行为投影为实际软件，而不是把文档完成当成模块完成。以领域操作、状态、不变量、事件、外部保证和关键未知为依据，选择真正需要的技术设计与验证：

- 输入输出需要跨进程或跨系统时，明确接口、协议、兼容和失败语义；
- 领域命令模型已经明确会读取和改变哪些状态与事实后，再结合业务查询、历史、审计、恢复、迁移和运行要求设计数据、存储、缓存与一致性；
- 多操作需要协调时，明确应用用例、事务、并发、幂等、超时、补偿和恢复；
- 人机交互会改变业务理解时，形成可验证的交互或原型；
- 平台、设备、外部接口或性能能力可能推翻方案时，按 [validation.md](validation.md) 在仓库根目录 `验证/` 建立并实际运行针对性的 PoC 或最小端到端验证，再把结论与适用条件回写任务和所属设计；
- 权限、隐私、审计、部署、成本、可观察性或兼容性会影响领域保证时，形成相应设计；
- 已进入实现时，建立领域操作、设计、代码与风险相称测试之间可追溯的对应。

这些是按问题选择的技术投影，不是每个模块必须填写的清单。数据库设计只在领域命令流程已经明确状态变化、并且系统确有保存或查询需要时出现，以满足当前需要的最小结构开始；Domain 本身不声明持久化。页面原型只在交互和功能假设需要观察时出现；PoC 只验证可能推翻领域或技术路线的命题。技术投影可以暴露 Domain 设计不可实现、模块边界不合理或业务目标有误，出现这类证据时返回真正受影响层次，不在技术文件里另立一套领域语义。

一个模块达到当前任务所需的落地完成条件，需要同时满足：职责和完整行为已可见，当前 Domain 的对象—命令—操作—内部状态变化—结果—事件已经自洽，必要的跨域协作与技术投影已经在各自位置形成，用户要求的实际实现已经完成，关键结果经过风险相称的验证，权威图、说明、设计与代码没有已知语义冲突。未进入当前任务范围的能力可以保持开放，不要把“本轮模块落地完成”写成模块永久封闭。

## 跨系统关系与接口归属

不建立固定的 `systems/collaboration/`。跨系统内容按它实际回答的问题归属：

- 为了完成总体业务目标而经过多个系统的业务路线，保存在 `business/`；
- 系统之间的高层依赖、责任交接和上下文关系，保存在 `systems/index.md` 或相关系统入口；
- 某个 Domain 接收的命令、查询条件、外部事实以及产生的结果和事件，由所属模块的 `领域.md` 定义自身语义；调用来源、消费者、传输契约和交付方式在应用协作或接口设计中维护；
- 使用方在自己的模块认知中记录依赖、前提和失败影响，通过链接引用提供方契约，不复制一份竞争定义；
- 真正负责跨系统推进、补偿、超时和恢复的协调者，应被建模成有明确所有者的应用模块或流程管理器，而不是塞进任一业务 Domain 或放进匿名“协作”目录。

不同限界上下文之间需要上下文映射或防腐层时，把关系保存在最接近其所有者的系统或模块材料中。一个接口、事件或结果只能有一个权威定义位置；跨系统时序图可以辅助解释，但不能脱离拥有行为和契约的模块成为独立事实副本。

## 宏观搭配与端到端检查

最初的系统全局编排用于指导模块拆分，是带有证据和假设的整体方案；模块落地后的宏观检查用于证明这些模块组合起来仍然实现需求目标和业务结果。两者不能合并成一次纸面确认，也不能因为每个模块单独正确就推断整体一定正确。

在重要模块形成完整草案、技术验证改变边界、完成一组实现或准备交付时，从上到下检查：

1. **目标覆盖**：每个当前大目标和收益由哪些业务目标线承载，是否仍存在没有路线的目标或不服务目标的系统能力；
2. **业务承接**：每条关键业务路线的参与者、判断、结果、异常和结束状态是否被明确模块操作承接；
3. **责任闭合**：系统和模块的责任、非责任、状态所有权、判断主权及对外保证是否无空缺、无冲突、无重复实现；
4. **契约兼容**：提供方和使用方对输入、输出、事件事实、顺序、版本、失败和幂等的理解是否一致；
5. **因果闭合**：各 Domain 内部命令怎样形成状态变化和领域事件，外部业务编排怎样将这些独立影响组合到明确业务结果；
6. **数据闭合**：数据设计是否覆盖实际查询、历史、审计、恢复和运行需要，每项持久化结构是否能追溯到领域状态变化或明确系统要求；
7. **技术成立**：决定性接口、数据、一致性、外部能力、性能、安全、部署和可观察性假设是否已经得到足够证据；
8. **真实结果**：组合后的实际系统能否通过代表性端到端路线产生目标要求的结果，并避免已经明确的不可接受结果。

使用需求目标与业务目标线覆盖图、跨系统或跨模块时序图、状态图和事件因果图完成纸面检查，再根据风险选择契约核对、最小端到端实现、真实运行、人工操作或自动测试。已确认的宏观图在其表达范围内成为当前权威；任何模块图、设计、代码或运行行为与它冲突时必须显式解决。

宏观检查不是只在所有模块完成后执行的一次最终门禁。每完成一组会相互协作的模块即可检查并尽早返回修正；最终交付前再检查当前目标范围是否整体闭合。若问题来自需求目标、业务路线、模块边界、领域语义、技术方案或实现缺陷，分别返回对应层次，不把所有失败都归为集成问题。

## 从任务提升系统认识

系统或模块新增、拆分、合并、改责，以及领域或设计变化，首先在 `work/` 记录变化目标、理由、备选、证据与影响。`work/` 不承担模块正文：系统和模块地图已经成立时就建立或更新 `认知.md`；已经有依据的对象、命令、操作、内部状态影响、领域结果和事件持续写入 `领域.md`；跨域业务路线、应用协作和技术投影分别写入它们自己的权威位置。尚未解决的竞争方案留在任务中，并在模块认知中标明受影响范围。

候选获得共同接受且足以指导后续工作时：

1. 更新受影响的系统全景、系统入口、模块 `认知.md`、`领域.md` 或按需设计文件；
2. 检查相关 Domain 是否各自完整承接应负责的输入、命令、内部操作、对象影响、结果、事件和异常语义，并在业务层检查它们怎样组合；
3. 若业务意义变化，同时更新 `business/`；若没有，明确这只是系统实现变化；
4. 在任务中记录已经提升到哪些当前基线，并保留必要证据和未解决限制；
5. 检查其他进行中任务是否依赖旧系统认识。

任务中“正在考虑拆分系统”是工作事实；只有被接受并写回 `systems/` 后，“系统已经这样划分”才成为当前系统事实。代码已经存在某种结构只能作为证据，不能跳过判断自动改写系统基线。

若用户纠正改变业务目标、系统存在目的、核心主线、模块边界或 Domain 所有权，所有依赖旧前提的下层材料立即降级为 `待重新判断` 的候选，并在任务中记录影响范围。先返回真正受影响的层级重建权威图和文字，再重新推导领域、协作、数据和实现；不得把旧设计视为沉没成本，仅做局部改名或补丁。

## 拆分与合并条件

在以下情况拆出独立系统、模块或模块内设计文件：

- 有明确且稳定的独立责任；
- 会被多个操作、模块、系统、业务线或任务引用；
- 具有独立变化频率或需要不同参与者理解；
- 单一材料已经无法清楚表达边界、领域行为或设计；
- 不拆分会迫使新参与者读取大量无关内容。

在内容很少、始终共同变化、无法形成独立责任或拆分后只能重复上级说明时保持合并。不要把每项能力或每张图自动拆成文件；目录整齐不是拆分理由。

## 常见机械错误

- 在读取现有资料、代码、原型或运行证据前先问用户，让用户替 AI 查找和总结。
- 把“请用户纠偏”变成逐题问答，只提供问题而不提供完整解释、推荐和依据。
- 给出多个选择却不作分析和推荐，把能够由 AI 承担的判断全部退回给人。
- 把“完整”理解为把系统、模块、Domain、异常、表和接口全部列出，而不是完整回答当前层的唯一核心问题。
- 把业务价值核心、依赖核心和风险核心混为一谈，因为某个内部机制复杂、被大量依赖或容易失败，就让它取代用户核心业务成为文档主角。
- 按调查、依赖、风险或实现顺序直接写系统文档，读者先看到版本、状态、对账、审计和异常机制，却不知道谁为什么进入系统。
- 系统入口一上来就是模块地图、状态所有权和接口关系，没有先说明现实背景、系统目的、核心业务主线及直接结果。
- 把核心业务和支撑闭环平铺成同等重要的模块或能力，使正常主线被停用、替换、恢复和治理细节淹没。
- 只建立一份业务全局认识，再把它裁短复制到系统和模块，导致下层没有说明自己独立承载什么能力、目标、操作和结果。
- 从 `business/index.md` 或 `systems/index.md` 直接跳到模块与领域细节，省略单个系统自己的认知入口。
- 把业务层的核心与辅助分类永久下传，不重新判断某项上层支撑能力是否正是某个系统或模块的核心职责。
- 系统或模块入口只描述“这是什么”和静态边界，没有说明核心做什么、怎样产生结果、目前到哪里以及接下来应进入什么。
- 用问题地图、待确认清单、模块定义或能力名称代替“系统 → 模块 → 模块要完成的事情、结果与内部能力”编排，用户始终看不见完整系统实际怎样运行。
- 每轮只确认一个判断并承诺下一轮再讨论下一项，虽然材料已经足以形成完整草案。
- 按现有系统边界一一生成业务线，使端到端业务结果被技术结构切碎。
- 把业务目标线目录嵌套在某个系统下，使跨系统业务无权威位置。
- 把仓库、进程、服务、页面区或代码包自动当成系统或模块。
- 在每个系统文件中复制同一业务规则，修改后形成多个版本。
- 按业务目标线给系统建立一套功能目录，使同一能力被重复实现或描述。
- 把 `认知.md` 写成职责简介，没有主要操作、参与者、输入输出、结果和覆盖状态。
- 把 `领域.md` 写成对象关系或 DDD 名词表，没有命令处理、内部对象变化、领域结果和完成事件。
- 在生产领域中维护“谁会消费这个事件、消费后执行什么”，使一个 Domain 开始拥有其他 Domain 的内部逻辑。
- 把多个 Domain 的先后、并行和条件路线写进其中一个 `领域.md`，让局部模型冒充宏观业务编排。
- 因为没有事件系统就把直接调用当成领域语义；直接调用只是当前执行方式，不能改变两个 Domain 各自的规则所有权。
- 在 `领域.md` 中出现 API、页面、数据库表、消息队列或具体调用方，并让这些技术或外部形态决定领域命令和对象。
- 只因对象有状态或生命周期就创建子域，没有独立问题、语言、操作和规则。
- 创建 `domains/` 或 `subdomains/`，而不是固定中文目录 `子域/`。
- 将对象提升为子域后，父领域仍把它定义为聚合内部实体，子域又把它定义为独立聚合根，形成两份竞争权威。
- 只移动文件而不重新检查聚合、一致性、事务、事件和失败恢复边界。
- 把图当成可有可无的插图，只写长篇文字，没有通过图让用户确认模块全景和事件因果关系。
- 将未确认候选混入权威图，或让文字、图和设计分别维护三套互相冲突的关系。
- 只列模块名称和接口，不说明操作前提、判断、状态影响、结果、不变量和失败语义。
- 把输入输出误判为实现细节，在尚未决定 JSON、HTTP 或消息格式时连领域语义契约也不写。
- 固定建立 `capabilities/`，把一个模块拆成若干无法独立指导开发的能力摘要。
- 固定建立 `collaboration/`，重复业务路线或模块接口，却没有明确行为和契约所有者。
- 把跨系统时序图当成业务路线图，掩盖参与者意图和业务结果。
- 在业务材料中写满接口、表和调用顺序，使当前技术方案变成业务定义。
- 把任务候选直接写入 `systems/`，让未接受方案看起来已经生效。
- 以“还有局部未确认”为由把全部模块正文留在 `work/`，使 `systems/` 只剩职责和占位语句。
- 只记录系统负责什么，不记录非责任、依赖条件和对外保证。
- 业务主线一确认就直接画表，没有先形成模块的领域命令地图和命令处理因果链。
- 根据名词或对象清单一对象一表，命令会读取、改变和保证什么尚未明确，只能靠数据库经验补造关系。
- 把表结构、ORM 或现有数据库反向提升为 Domain 真相，让存储便利静默改变命令、聚合、不变量或事件。
- 将所有过程数据永久保存，或预建通用扩展字段、关系和未来表，却无法指出它服务哪个领域状态、事实、恢复或查询需要。
- 把“哪些数据需要保存、保持多久、供谁查询以及怎样恢复”写进 `领域.md`，让 Domain 开始承担存储和外部使用责任。
- 完成 `认知.md` 和 `领域.md` 就宣布模块落地完成，没有形成必要技术投影、实际实现和结果验证。
- 每个模块分别看起来正确就宣布系统完成，没有重新检查需求目标、业务路线、契约、事件因果和失败恢复能否端到端闭合。
- 为每个代码组件预建模块文件，制造无法维护的结构镜像。
- 要求开发者重新阅读全部业务文档，才能知道某项系统能力必须怎样实现。
- 因文件存在、链接有效、Mermaid 可渲染、表格无缺列或 DDD 术语齐全，就宣布当前认识通过；机械校验只能证明材料结构可用，不能证明语义正确或已被接受。

## 必须保留的语义结果

按项目实际复杂度保留：现实系统背景与建设边界、系统存在理由、核心业务主线与直接结果、各主线的支撑闭环、已确认的系统与模块全景图、核心业务主线—模块操作覆盖关系、每个系统的责任与非责任、模块与子域地图、每个模块及已提升子域的 `认知.md`、已确认的对象—命令—操作—内部状态—结果—事件视图、父子领域语义所有权、领域自身的语义输入输出、不变量、关键失败与恢复、业务层的跨域编排、系统间必要依赖与接口所有者、应用协作选择、领域状态变化与数据设计的追溯关系、按需设计、领域与实现映射、端到端验证证据、当前实现偏差、业务语义来源和任务入口。

最终检查分三层：一个首次接触项目的人只读取系统入口，就能说明系统处在什么环境、为谁完成哪些核心事情以及模块怎样承载；开发者读取某个模块的 `领域.md`，能独立实现该 Domain 接收的命令、内部判断、对象变化、结果、事件和失败语义，而不需要知道调用来源、消费者或存储方式；继续读取相关业务路线和设计后，才知道多个 Domain 为什么以何种顺序组合、事件或直接调用怎样执行、哪些状态需要怎样保存，并能检查实现没有让外部编排或技术投影污染 Domain。
